## 二、向量检索、混合检索与 RAG 架构

### 1. 什么是向量检索？基本原理是什么？

向量检索是把文本、图片等数据通过 embedding 模型映射成高维向量，再用余弦相似度、内积或 L2 距离找语义相近内容。核心流程是：离线/异步建索引，把消息、附件、记忆转成 embedding；查询时把用户 query 也转成向量；在向量库中做 ANN 近似最近邻搜索；最后把命中的内容作为上下文注入模型。

本项目里，消息和 memory 都有 Qdrant 向量检索支持：消息侧在 `internal/backend/agentruntime/message_vector_index.go`，记忆侧在 `internal/backend/agentruntime/memory_vector.go`。默认距离是 Cosine，集合不存在时会自动创建。

### 2. 向量检索相比关键词检索有什么优势？本质区别是什么？

关键词检索匹配的是字面词项，比如 BM25/倒排索引更擅长精确词、代码名、ID、错误码。向量检索匹配的是语义距离，能处理同义表达、模糊意图和自然语言问题。

本质区别是表示空间不同：关键词检索在稀疏词项空间做匹配，向量检索在稠密语义空间做近邻搜索。项目里因此没有押注单一路线，而是支持 SQL fallback、Elasticsearch/OpenSearch 全文检索、Qdrant 语义检索和 hybrid 模式。

### 3. 高维向量检索的性能瓶颈一般在哪里？

主要瓶颈有四类：

- embedding 生成耗时和成本，尤其是批量消息、附件分块和 memory 重建时。
- ANN 索引查询成本，高维向量越多，召回和延迟之间越需要取舍。
- 网络与外部服务延迟，Qdrant/embedding provider 都可能成为链路瓶颈。
- 结果 hydration，把向量命中的 ID 回查成完整消息、权限过滤、会话过滤时也会增加耗时。

本项目的做法是：检索服务有超时配置；消息附件会按 chunk 建索引；Memory 向量检索会先过滤 `user_id`、`status`，再回查 memory item；检索失败时还能回退到基础 memory context，而不是直接影响主对话。

### 4. 为什么要做混合检索，而不是只用纯向量检索？

纯向量检索容易漏掉精确约束，例如错误码、函数名、文件名、订单号、配置 key。纯关键词检索又不擅长语义改写和模糊需求。因此混合检索适合 Agent 场景：用户既可能问“上次那个登录 bug”，也可能搜“AGENT_API_JWT_SECRET 配置在哪”。

项目里的 `MessageSearchService` 支持 `hybrid` backend：关键词侧走 Elasticsearch/OpenSearch，语义侧走 Qdrant，最后用 RRF（Reciprocal Rank Fusion）合并排序。

### 5. 项目里怎么混合使用向量检索和 BM25/全文检索？

可以这样讲：

> 我们把检索分成 keyword、semantic、fallback 三层。keyword 使用 Elasticsearch/OpenSearch，适合 BM25/全文匹配；semantic 使用 Qdrant + embedding，适合语义召回；fallback 是 SQL/文件存储里的简单 contains 搜索。  
> hybrid 模式下，两路各取一个窗口，然后用 RRF 融合，避免手工调两个分数的绝对尺度。代码里 `rrfMergeMessageSearchResults` 会按 rank 加权，默认 `RRFK=60`。

### 6. 什么情况下优先向量检索，什么情况下优先 BM25？

优先向量检索：

- 用户问题是自然语言意图，例如“我之前说过偏好什么风格？”
- 同义改写多，字面词不稳定。
- Memory、长对话、知识片段这类语义相关性更重要的场景。

优先 BM25/全文：

- 查精确 ID、函数名、错误码、配置项、日志关键字。
- 查询词本身非常明确。
- 需要可解释的高亮、片段定位。

本项目实际选择是产品层暴露统一搜索接口，后端通过配置选择 `sql`、`elasticsearch`、`semantic` 或 `hybrid`，避免把策略写死。

### 7. 整体检索架构是怎么设计的？

可以按链路讲：

1. 写入侧：用户消息、附件文本、memory item 写入主存储后，异步或生命周期 worker 触发索引。
2. 索引侧：全文索引写 Elasticsearch/OpenSearch；向量索引写 Qdrant；附件会切块。
3. 查询侧：统一进入 `MessageSearchService` 或 `MemoryVectorService`。
4. 融合侧：hybrid 用 RRF 合并；memory retrieval 也会融合向量召回和关键词/权重打分。
5. 注入侧：只把用户可见、状态 active、会话可见的结果注入上下文，避免跨用户和跨会话污染。
6. 降级侧：外部检索不可用时回退到 SQL/fallback 或基础 memory context。

### 8. RAG 架构与检索流程怎么完整讲？为什么引入父子索引？

可以把 RAG 讲成“离线建索引 + 在线召回排序 + 上下文工程 + 评测闭环”四层。

离线侧先做文档解析、清洗、切块、metadata 提取、embedding、全文索引和向量索引。在线侧收到 query 后，先做用户/租户/会话权限过滤，再并行跑 BM25、向量检索、memory 检索、最近会话或业务 API 检索，随后做融合、rerank、token budget 选择和 prompt 拼接。

父子索引的核心原因是解决“检索粒度”和“阅读粒度”不一致：

- 子块适合检索：粒度小，embedding 更聚焦，召回更准。
- 父块适合注入：保留标题、上下文、表格前后文、代码所属函数/文件，减少模型误解。
- 多个子块命中同一父块时可以合并去重，避免 prompt 里重复塞相邻片段。

面试里可以这样说：

> 我们不会直接把整篇文档向量化，也不会只用很碎的 chunk。索引时保留 parent-child 关系，召回时用子块提高命中率，注入时回到父块或父块摘要，保证模型拿到可读、完整、少重复的证据。

### 9. 为什么检索阶段还要引入 BM25？BM25 和向量检索怎么组合，比例如何设置？

BM25 的价值是补向量检索的短板。向量检索擅长语义相似，但对函数名、错误码、配置 key、商品 ID、日志关键字、专有名词不稳定；BM25 基于词项匹配，更适合精确约束、可解释高亮和关键词定位。

组合方式一般有三种：

- 并行召回 + RRF 融合：两路各取 topN，只看排名不看绝对分数，工程上最稳。
- 加权分融合：归一化后按 `score = alpha * vector + (1 - alpha) * bm25`，适合分数校准过的系统。
- 分场景路由：代码/ID/错误码 query 提高 BM25 权重，自然语言意图提高向量权重。

比例不要拍脑袋。可以先用默认权重做 baseline，比如向量 0.6、BM25 0.4；如果业务更偏精确搜索，就反过来 BM25 0.6、向量 0.4。更推荐用 RRF 起步，因为 BM25 分和向量相似度不是同一量纲。

本项目里 message search 的 hybrid 模式使用 RRF，默认 `RRFK=60`。这样避免手工调两个分数的绝对比例，后续可以通过 golden set 观察 Recall@K、Context Precision、延迟和成本，再决定是否对特定 query 类型做动态权重。

### 10. 从 carry 到最终上下文的完整流程是什么？

可以把 carry 理解成一次 Agent turn 里从上游传下来的“可用信息包”，包括用户消息、session、用户 profile、memory、附件、工具结果、检索候选和 token budget。完整流程是：

1. 入口归一化：拿到 user query、session id、tenant/user 权限、附件和当前任务目标。
2. Carry 构建：加载最近消息、session summary、用户长期 memory、显式偏好、工具状态和预算。
3. Query 理解：识别是否需要 RAG、工具、澄清、写操作或高风险审批。
4. 多路召回：BM25、向量、父子索引、memory、最近会话、业务 API 并行查询。
5. 权限过滤：按 user、tenant、session、ACL、status、visibility 过滤。
6. 融合与 rerank：RRF/加权融合后，对候选做轻量或 cross-encoder rerank。
7. Context selection：按 token budget 选择 evidence，去重、合并父块、保留来源。
8. Prompt 拼接：系统约束、开发者约束、用户目标、历史摘要、短期上下文、长期 memory、RAG evidence、工具说明、输出格式。
9. 生成与后处理：模型输出后做引用/格式/安全校验，必要时工具调用或 repair retry。
10. 回写：消息落库、trace、usage、evaluation、memory candidate 和缓存失效。

面试重点是：carry 不是把所有历史都塞进去，而是作为候选池；最终上下文必须经过过滤、排序、压缩和预算分配。

### 11. 检索阶段有没有 rerank？用什么方式，rerank 后返回几个块？

有。召回阶段追求高召回，所以会拿一个相对大的候选窗口，例如 30 到 100 个 chunk；rerank 阶段再追求相关性和可用性。

常见 rerank 方式：

- 轻量规则 rerank：词项重合、标题命中、metadata 权重、时间新鲜度、原始召回排名。
- Cross-encoder rerank：把 query 和 chunk 拼成 pair，由 rerank 模型输出相关性分数。
- LLM-as-reranker：适合复杂多跳或高价值请求，但成本和延迟更高。
- Hybrid rerank：先规则粗排，再对 topN 用模型重排。

一般不会把 rerank 后的所有结果都塞给模型。经验值可以是：候选召回 top 50，rerank 后注入 top 5 到 10 个块；如果父子索引回父块，最终父块数量通常控制在 3 到 6 个。这个数量的原因是：足够覆盖主要证据，又不会让无关上下文稀释注意力、推高 token 和延迟。

本项目已经支持轻量本地 rerank：对 RRF/语义召回候选按 query 词项匹配、原始融合分和时间新鲜度重排；候选窗口可通过 `AGENT_API_MESSAGE_SEARCH_RERANK_CANDIDATE_LIMIT` 控制。

### 12. Rerank 后 topK 阶段怎么做？为什么是这个值，有没有尝试过其他策略？

topK 不是固定神秘数字，而是 token budget 和评测共同决定的策略。

可以这样答：

> 我们先设一个保守默认值，比如召回 top 50、rerank top 8、最终注入 5 个左右证据块。然后用 golden set 看 Recall@K、Context Precision、Faithfulness、端到端延迟和 token 成本。如果 top 5 经常漏证据，就提高到 top 8 或引入父块摘要；如果 top 10 带来噪声，就降低注入数或加 score threshold。

尝试过的策略可以包括：

- 固定 topK：简单稳定，适合 MVP。
- 动态 topK：按 query 模糊度、rerank 分数断崖、任务类型调整。
- Score threshold：低于阈值不注入，避免硬塞弱相关证据。
- MMR 去冗余：降低相似 chunk 重复，提升证据多样性。
- Parent merge：多个子块命中同一父块时合并成一个父上下文。

面试里要强调：最终看评测，不是“我觉得 5 个差不多”。

### 13. 上下文长度不够或过长怎么处理？上下文工程怎么设计？

上下文工程的核心是“预算分配 + 信息分层 + 可追溯压缩”。结构通常是：

1. System/developer 约束：安全、角色、工具规则。
2. 当前用户目标：最新 query 和任务边界。
3. 会话摘要：压缩后的长期对话状态。
4. 最近消息窗口：保留最近几轮细节。
5. 相关 memory：用户偏好、事实、长期项目背景。
6. RAG evidence：带来源、标题、时间和权限信息的证据块。
7. 工具说明与输出格式：只放当前可用工具和必要 schema。

上下文过长时，处理顺序是：

- 先删无关内容：低分 evidence、重复 chunk、过期 memory、不可见内容。
- 再压缩长内容：父块摘要、会话摘要、工具结果摘要。
- 保留最近关键交互和高置信 evidence。
- 对超长附件按需检索，不直接全量注入。
- 如果仍不够，让 Agent 分轮检索或请求用户缩小范围。

避免长上下文导致模型性能下降，关键是减少噪声而不是单纯追求大窗口。无关 evidence 会稀释注意力，甚至诱导幻觉；所以要用 rerank、score threshold、MMR、metadata 过滤和 token budget，让最终上下文“少但准”。

## 三、RAG 评测与效果验证

### 1. 你是怎么验证模型效果的？RAG 具体怎么做评测？

不要只说“回答正确率”。可以分三层：

- **检索层**：Recall@K、MRR、命中文档覆盖率、是否命中 gold evidence。
- **生成层**：答案准确性、引用支撑度、是否遗漏关键约束、是否幻觉。
- **线上运行层**：工具错误率、LLM 失败率、首 token 延迟、P95 延迟、成本、空输出率、风险事件。

本项目已有 `EvaluationEngine`，会基于真实运行 trace 统计 pass/warning/failed、tool error、LLM failure、latency、token、cost、risk、empty output 等指标，代码在 `internal/backend/agentruntime/evaluation_engine.go`、`evaluation_metrics.go`、`evaluation_rules.go`。

项目改造状态：

- [x] 已补齐 Golden Set / RAGAS 风格评测入口：新增 `internal/backend/agentruntime/evaluation_golden.go`。
- [x] 已支持 answer correctness、answer relevancy、faithfulness、context precision、context recall 五类指标。
- [x] 已支持 `GoldenJudge` 接口，并通过 `AGENT_API_EVAL_JUDGE_ENABLED`、`AGENT_API_EVAL_JUDGE_MODEL` 和 `AGENT_API_LLM_MODEL_ROUTES` 的 `judge=...` 路由绑定线上 LLM-as-Judge 模型；默认仍可用启发式 judge 保证 CI 稳定。
- [x] 已支持 `AGENT_API_EVAL_JUDGE_PROMPT_VERSION` 管理 judge prompt 版本，并把 prompt version 写入 LLM-as-Judge 结果 metadata，便于排查分数漂移。
- [x] 已把失败或低置信样本接入 `EvaluationReview`，进入人工复核闭环。
- [x] 已添加回归测试：`internal/backend/agentruntime/evaluation_golden_test.go`。

### 2. 评测是不是只判断“回答是否正确”，有没有中间态？

应该强调有中间态。RAG 评测至少要拆成：

- query 是否被正确理解。
- 检索是否召回正确 evidence。
- rerank/融合是否把正确 evidence 排到前面。
- prompt 是否正确使用 evidence。
- 生成答案是否忠实、完整、可追溯。

项目评测也不是只有正确/错误，而是 `passed`、`warning`、`failed` 三类；例如空输出、LLM failure 是 warning，工具错误、高风险事件是 failed。

### 3. 简历里写的“准确率 96%”怎么统计？

如果要写 96%，建议这样解释：

> 这是在固定离线评测集上的通过率，不是无条件线上准确率。评测集按业务场景分层，例如记忆问答、会话搜索、工具调用、附件问答、多轮上下文。每条样本有标准答案或关键断言，先自动判分，再抽样人工复核。分母是有效样本数，分子是自动和人工都通过的样本。

如果当前项目没有真实 96% 数据，面试时不要硬说。可以改成：“我们用 pass rate 评估，目前项目里评测框架已支持真实 trace 汇总，但具体业务准确率需要基于线上数据持续校准。”

### 4. 这个 96% 可靠性、偏差有验证吗？

回答重点：

- 做 bootstrap 置信区间或按场景分桶统计，避免平均值掩盖 badcase。
- 保留人工复核集，检查自动 judge 的偏差。
- 区分离线集、灰度集、线上真实用户集。
- 对低频长尾场景单独看，不用总体准确率代表全部。

可以补一句：本项目评测更偏真实运行 trace 的工程指标，真正业务准确率还要叠加人工标注和线上反馈闭环。

### 5. 真实用户使用时效果是否真的能达到 96%？

不能直接承诺。应该说：

> 离线 96% 只能说明在覆盖到的样本分布上表现好。真实用户 query 更发散，会出现冷启动、上下文缺失、附件质量差、工具失败、时效性问题，所以线上一般要低于离线。我的做法是把线上日志、用户反馈、工具错误和高风险事件纳入评测，再反向补充评测集。

### 6. 有没有考虑更接近真实使用场景的数据做最终校验？

有，应该用：

- 真实会话 trace 回放。
- 用户反馈和人工审核样本。
- 工具调用链路样本，包括失败重试。
- 长上下文/多轮对话样本。
- 冷启动用户和高活跃用户分层样本。

本项目的评测源就是 runtime trace，可以覆盖 jobs、skills、LLM usage、risk events、artifacts 等真实链路数据。

### 6A. RAG 评测数据集一般包括什么？怎么构建高质量评测数据？

RAG 评测集不要只有 query 和标准答案，最好包含：

- query：用户原始问题，保留口语、省略、指代和多轮上下文。
- expected answer：标准答案或关键事实断言。
- gold evidence：必须命中的文档、chunk、memory 或工具结果。
- context：多轮历史、用户权限、时间、业务场景。
- negative cases：无答案、权限不足、证据冲突、过期知识、诱导幻觉。
- metadata：场景标签、难度、数据版本、prompt/model/index 版本。

构建方式：

1. 从真实 trace、客服问题、线上搜索日志和 badcase 中抽样。
2. 按场景分桶：FAQ、配置、故障排查、长文档、多轮、工具调用、权限。
3. 人工标注 gold evidence 和必须命中的事实，而不是只写一个模糊答案。
4. 加入 hard negative：字面相似但答案不同、同名不同版本、无权限内容。
5. 固定版本，改检索、rerank、prompt、模型时在同一集合上回放。

常见指标可以分成检索和生成两类：检索看 Recall@K、MRR、NDCG、Context Precision/Recall；生成看 answer correctness、faithfulness、answer relevancy、引用支撑度、幻觉率。线上再看延迟、成本、空回答率、用户反馈和人工复核通过率。

### 6B. 如果要提升相关度和回答效果，优先优化检索阶段还是生成阶段？怎么验证有效？

先做 badcase 归因，不要一上来调 prompt。

- 没召回 gold evidence：优化检索阶段，比如 chunk、metadata、query rewrite、BM25/向量融合、topK、rerank。
- 召回了但排得靠后：优化融合排序、rerank、score threshold、MMR。
- 证据进了上下文但回答错：优化 prompt、上下文结构、引用要求、模型选择或生成后校验。
- 证据冲突或过期：优化数据治理、版本、时间优先级和可信源策略。
- 工具数据更权威：从 RAG 改成 tool/API 查询，不让模型猜静态知识。

验证方式是固定评测集做 A/B：同一批 query、同一模型、同一 prompt，只改一个变量。看 Recall@K、Context Precision、Faithfulness、端到端 pass rate、P95 延迟和 token 成本。线上再小流量灰度，看用户反馈、人工复核和风险事件。

### 7. RAG 和微调怎么取舍？

可以先给结论：RAG 和微调不是互斥关系，解决的问题不同。

RAG 更适合：

- 知识更新频率高，例如业务规则、商品、文档、会话历史。
- 数据有隐私或权限边界，必须按用户、组织、会话做隔离。
- 需要可解释性和可追溯，答案最好能回到 evidence。
- 希望低成本快速上线，不想每次知识变化都重新训练。

微调更适合：

- 模型不会稳定遵循某种输出格式或工具调用风格。
- 需要学习特定语气、领域表达、分类边界或固定操作习惯。
- 任务分布稳定，有高质量训练样本和评测集。

结合本项目，我会优先把外部知识、用户 memory、会话历史放在 RAG/检索层；把“回答风格、工具调用格式、拒答边界、结构化输出习惯”留给 prompt、schema 或微调。真实系统通常是 RAG 负责“知道什么”，微调负责“怎么说、怎么做”。

### 8. 多路召回怎么做？

多路召回的目标是弥补单一路径的盲区。向量检索擅长语义相似，但对专有名词、错误码、函数名、配置 key 不一定稳定；BM25/倒排索引擅长字面匹配，但对同义改写不够友好。

常见做法：

1. Query 预处理：保留原 query，同时抽取关键词、实体、代码符号、时间范围。
2. 多路召回：向量召回、BM25/全文召回、SQL 过滤、热门/最近内容、用户 memory 召回并行执行。
3. 权限过滤：先按 `user_id`、session、visibility、status 过滤，避免召回不该看的内容。
4. 融合排序：用 RRF、加权分数或 reranker 合并结果。
5. 上下文压缩：只把 top evidence 的关键片段注入模型。

本项目的 message search 就支持 SQL fallback、Elasticsearch/OpenSearch、Qdrant semantic 和 hybrid；hybrid 下用 RRF 融合，memory retrieval 也会把向量召回和关键词/权重召回合并。

### 9. 如何降低 RAG 延迟？

可以分链路优化：

- 检索前：query rewrite 不要过重，能规则化就不要每次调大模型。
- embedding：缓存 query embedding；批量生成文档 embedding；选择合适维度而不是盲目追大模型。
- 向量库：使用 HNSW/ANN 索引，合理设置 `ef_search`、topK、score threshold。
- 多路召回：并行召回，设置每路 timeout，慢路径超时降级。
- 上下文：减少 topK 和 chunk 数，先 rerank 再注入，避免把大量无关文本塞进 LLM。
- 系统层：热点 query、热门文档、用户 profile 做缓存。

结合本项目可以说：搜索服务有 timeout，Qdrant/Elastic 不可用时可以 fallback 到 SQL 或基础 memory context；长耗时任务走 job/SSE 事件流，不把用户请求卡死在同步 HTTP。

### 10. 如何解决幻觉问题？

不要只说“prompt 写不要编造”。更完整的答案是：

- 检索侧：提高 evidence 召回质量，降低无关上下文；必要时要求答案引用来源。
- Prompt 侧：明确“只能基于上下文回答，不知道就说不知道”，并规定冲突证据处理方式。
- 解码侧：降低 temperature/top-p，减少创造性输出。
- 工具侧：需要事实时优先查数据库/API/搜索，而不是让模型猜。
- 校验侧：生成后做 groundedness 检查、格式检查、事实一致性检查。
- 评测侧：把 hallucination badcase 加入回放集，按场景统计。

本项目里，memory 只注入 active/visible/relevant 内容，检索失败可以降级；评测 trace 会记录空输出、工具错误、风险事件和 LLM failure，方便定位幻觉到底来自检索、工具还是模型本身。

### 11. 如何让模型输出稳定格式？

优先级从工程上应该是：

1. 使用模型原生 structured output / JSON schema / function calling。
2. Prompt 明确输出格式，并给 few-shot。
3. 生成后做 JSON/schema 校验，失败则带错误信息 repair retry。
4. 对非常稳定、长期存在的格式任务，再考虑 SFT/LoRA。

面试里可以强调：不要只依赖 prompt。线上系统必须有解析、校验和重试，否则一次格式漂移就会把下游链路打崩。

### 12. LLM 调用很慢怎么办？

先拆指标：是 TTFT 慢，还是总耗时慢。

- TTFT 慢：减少首轮 prompt token、缩短检索链路、先返回状态事件、选择更快模型。
- 总耗时慢：限制 max output token、流式输出、长任务异步化、工具并行、结果缓存。
- 模型侧：小模型路由、量化/蒸馏、自部署推理服务、continuous batching。
- 产品侧：能 workflow 固化的逻辑不要每次都调 LLM。

本项目有 LLM governance：记录 latency、TTFT、token、cost、失败率；支持 timeout、重试、quota、熔断和多 backend 路由。Web 侧用 SSE/job event 让用户尽快看到进度。

### 13. 如何做缓存？

缓存要区分“确定性缓存”和“语义缓存”。

确定性缓存：

- key 可以由模型、prompt 模板版本、用户权限范围、检索版本、参数组成 hash。
- 用 LRU 控制容量，用 TTL 控制时效。
- 适合 FAQ、固定工具结果、embedding、热门检索结果。

语义缓存：

- 先把 query embedding 后和历史 query 做相似度匹配。
- 命中后不能直接无脑返回，最好再校验用户权限、时间有效性和答案是否仍适用。
- 对涉及个人数据、实时价格、订单状态、权限变化的答案要谨慎，TTL 要短或不缓存最终回答。

面试时可以说：我的缓存会分层做，embedding/query 级缓存更安全，最终答案缓存要把用户、权限、数据版本、prompt 版本都纳入 key；如果只是“相似问题直接返回旧答案”，很容易造成越权或陈旧答案。

项目改造计划：

- [x] 在文档中补充缓存体系改造计划，明确先做确定性缓存底座，再逐步接入 embedding、retrieval、prompt resolver，最终回答语义缓存默认不启用。
- [x] 新增统一缓存底座：`internal/backend/agentruntime/cache_system.go`，提供 `CacheStore`、`CachePolicy`、`TypedCache`、统一 key builder、内存/Redis/Noop 实现和基础命中率指标。
- [x] 新增统一配置入口：`AGENT_API_CACHE_BACKEND`、`AGENT_API_CACHE_REDIS_URL`、`AGENT_API_CACHE_PREFIX`、`AGENT_API_CACHE_DEFAULT_TTL`、`AGENT_API_CACHE_FAIL_OPEN`，并保留 message context、session list、live setup prompt 的细分配置覆盖能力。
- [x] 接入 prompt resolver cache：缓存 published prompt / experiment allocation，并用 cache-invalidating prompt store 在 prompt 发布、回滚、实验配置变化时清理 resolver namespace。
- [x] 接入 query embedding cache：key 由 provider、model、维度、task type、文本 hash 组成，消息语义检索、消息向量索引和 memory vector 共用这一层。
- [x] 接入 message search result cache：key 由 user、backend、query、分页、检索配置版本组成，`Runtime` 写入新消息后清理当前用户 search cache。
- [x] 接入 memory retrieval cache：key 由 user、session、query、embedding model version 和 retrieval 配置组成，缓存选中的 memory item；命中后仍会 hydrate 当前 item 并记录 injection，memory 写入、删除、设置变化后失效。
- [ ] 建立缓存观测：按 namespace 统计 hit/miss/error/latency，并在管理端或 readiness/metrics 中暴露。

## 四、Agent、Multi-Agent、Skill、MCP

### 1. 为什么选择 Multi-Agent，而不是 Single Agent？

回答不要绝对化：

> 我会优先用 Single Agent，只有当任务天然可拆、需要不同角色或可以并行验证时才引入 Multi-Agent。比如代码审查、需求分析、执行、验证可以拆成不同角色；但简单问答或单步工具调用没必要多 Agent。  
> 项目里同时支持 sub-agent、team/worktree 协调和普通单 Agent workflow，核心取舍是复杂度、上下文隔离、执行成本和可验证性。

### 2. Workflow 和 Autonomous Agent 怎么取舍？

- Workflow：流程确定、可控性高，适合注册、检索、上传、审核、固定工具链。
- Autonomous Agent：目标明确但路径不确定，适合代码修改、复杂排障、资料整合。

本项目里 Web API、任务队列、Memory、评测是 workflow；具体一次 Agent turn 内部可以自主选择工具、检索和子任务。

### 3. Agent 间怎么通信？

可以分三种：

- 共享状态：任务 ID、session、job event、artifact、memory。
- 消息通道：team/coordinator、Redis stream/job event、Go channel。
- 文件/工作区：worktree 或 artifact 作为交付物边界。

面试中可强调：通信不是越自由越好，关键是定义清楚输入、输出、终止条件和可观测事件，避免多个 Agent 同时污染同一上下文。

### 4. 怎么避免上下文污染？

项目里的做法可以概括为：

- 用户级隔离：Memory、session、search 都带 `user_id`。
- 会话级隔离：Memory 有 session visibility，检索时会过滤当前 session 可见性。
- 工具权限隔离：权限系统、sandbox、allowed tools。
- 上下文压缩：旧消息会被摘要替代，避免把无关历史无限注入。
- 检索过滤：只注入 active、visible、相关性足够的 memory。

### 5. Skill 和 MCP 是什么关系？

MCP 更像外部工具和上下文协议，解决“模型如何标准化接入外部系统”。Skill 更像任务能力封装，包含说明、触发条件、prompt、允许工具、执行策略。

本项目里 `internal/harness/skills/mcp_bridge.go` 可以把 MCP tool metadata 构造成 Skill，这说明二者可以桥接：MCP 提供工具接口，Skill 提供业务化使用方式和提示词包装。

### 6. MCP / function calling 有什么区别？

function calling 是模型 API 层的工具调用格式，通常解决一次模型调用如何输出结构化 tool call。MCP 是工具生态协议，关注 server、transport、tool/resource discovery、权限、生命周期和多客户端接入。

可以这样答：function calling 是“模型怎么叫工具”，MCP 是“工具怎么被发现、管理和接入”。

### 6A. Agent 设计和 Function Calling 怎么设计？

Agent 设计不要只说“让模型自己想”。生产里通常拆成四层：

- Planner：理解目标，决定是否需要检索、工具、澄清或直接回答。
- Tool Router：根据工具 schema、权限、上下文和置信度选择候选工具。
- Executor：执行工具，处理 timeout、重试、幂等、降级和审计。
- Verifier：检查工具结果是否满足任务，必要时继续调用、修正参数或停止。

Function Calling 设计重点是 schema 和控制权：

- 每个工具有明确 name、description、JSON schema、必填字段、枚举和边界。
- 服务端做参数校验，不能相信模型生成的参数天然安全。
- 工具分级：只读工具、低风险写工具、高风险写工具分别走不同权限和审批。
- 调用结果结构化返回，包含 success、error_code、retryable、summary、raw_ref。
- 每次调用写 trace，后续可以做评测、回放和故障定位。

可以这样答：

> 模型只负责提出 tool call intent，真正能不能执行由服务端决定。服务端会做 schema 校验、权限校验、敏感参数过滤、超时和审计。这样就算模型被 prompt injection 诱导，也不能越过工具白名单和权限边界。

### 6B. 任务规划是模型完成还是规则实现？多工具调用如何决定顺序？

我会做混合规划：确定性强的流程用规则或 workflow，路径不确定的任务交给模型规划。

规则适合：

- 登录、上传、检索、审核、固定报表。
- 明确的权限动作和高风险操作。
- 成本敏感、稳定可枚举的流程。

模型规划适合：

- 需求模糊、需要拆解目标的复杂任务。
- 需要根据中间结果动态选择下一步的任务。
- 代码修改、排障、多资料整合等开放任务。

多工具调用的顺序一般由依赖关系决定：先检索/读数据，再分析，再写入或执行；先低风险只读工具，再高风险写工具；能并行的只读工具并行，写操作串行并带幂等 key。

面试里可以补一句：Planner 可以给出 plan，但 Executor 必须验证前置条件。比如模型想调用“发送邮件”，服务端仍要检查收件人、内容、用户权限和是否需要二次确认。

### 6C. 工具调用失败、输入过滤和安全控制怎么做？

工具失败分三类处理：

- 参数错误：返回 schema validation error，让模型修正参数，重试次数有限。
- 临时错误：timeout、5xx、限流，按指数退避重试或切备用工具。
- 权限/安全错误：直接停止，不让模型绕路重试敏感接口。

输入过滤包括：

- schema 校验：类型、长度、枚举、必填字段。
- 业务规则：金额上限、批量数量、时间范围、资源归属。
- 安全校验：敏感词、注入指令、路径穿越、命令黑名单、外发域名限制。

敏感接口限制不能只靠 prompt。工程上要有工具白名单、RBAC/ABAC、租户隔离、scope token、只读默认、审批流、审计日志和沙箱。比如文件读取只能读工作区允许路径，shell 工具有命令/网络/超时限制，生产发布、删除、支付、发外部消息必须二次确认。

### 7. Prompt 怎么设计和迭代？有没有 Prompt 自动优化？

回答结构：

- Prompt 分层：系统约束、用户目标、工具说明、检索上下文、输出格式。
- 迭代方式：从 badcase 出发，定位是检索、工具、约束还是输出格式问题。
- 自动优化：可以用 eval trace 自动聚类失败样本，生成候选 prompt，再通过离线回放验证。

结合项目：Memory extraction、context summary、skills prompt wrapper 都是 prompt 工程的落点。比如上下文摘要 prompt 明确要求保留用户目标、关键事实、已做决定、未完成事项和工具结果。

项目改造状态：

- [x] 已有 Golden Set、LLM-as-Judge、Eval Review 和 Admin Evaluation，能支撑 prompt 离线回放。
- [x] 已有 WorkflowEngine，可以承载 badcase 聚类、候选生成、离线验证、人工 review 的自动优化流水线。
- [x] 已补齐 Prompt Registry：支持 prompt template、version、publish、rollback、diff、render preview，核心代码在 `prompt_registry.go`、`prompt_api.go`。
- [x] 已补齐 Prompt A/B：`PromptExperiment` 支持 running/paused/completed、variant 权重、按 user/session/job/request 做稳定分桶，LLM usage 会记录 `prompt_id`、`prompt_version`、`prompt_hash`、`experiment_id`、`variant_id`。
- [x] 已补齐 Prompt 自动优化入口：`prompt_optimization.go` 会基于低分/失败评测样本聚类生成候选版本，候选先进入 review/publish 流程，不直接替换线上 prompt。
- [x] 已补齐 Prompt resolver cache，并在 publish、rollback、experiment 变更时失效缓存，避免线上仍使用旧版本。

### 8. 用户请求不完整时，如何补全意图？

先做低风险推断，高风险分支再澄清：

- 从当前 session、最近消息、用户 memory、附件和项目上下文中补全。
- 如果缺的是可逆细节，可以先按默认策略执行。
- 如果涉及删除、发布、付费、权限扩大、不可逆操作，需要询问或走审批。

这也符合本项目权限设计：工具执行和风险操作不能只靠模型自由发挥。

## 五、Memory 设计

### 1. 长短期 memory 怎么设计？

本项目可以这样讲：

- 短期 memory：当前 session 消息、最近上下文、session memory、上下文摘要。
- 长期 memory：用户级 memory item，包括 fact、preference、event、skill 等类别。
- 检索注入：根据最后一条用户消息，从 memory item 中选取相关内容注入。
- 用户控制：支持 memory settings、删除、反馈、导出、维护。

对应模块包括 `internal/backend/agentruntime/store_sql.go` 的 memory service、`memory_vector.go` 的向量召回、`memory_governance.go` 的质量和冲突治理。

### 2. 记忆冲突怎么处理？

项目里不是简单覆盖，而是：

- 对 memory 做 raw hash 去重。
- 根据 namespace、category、level 判断是否可能冲突。
- 根据来源优先级、时间替代、置信度差异决定 candidate 胜出、旧记忆归档、候选归档或 pending confirm。
- 用户显式编辑的 personalization 优先级最高，模型抽取不能覆盖用户手动设置。

面试时可以举例：“用户之前说喜欢简短回答，后来手动设置喜欢详细回答，手动设置会优先，旧记忆归档或进入确认。”

### 3. Memory retrieval 怎么优化？

本项目用了混合思路：

- 向量召回：Qdrant 搜 active memory。
- 关键词/权重召回：按 query token overlap、session 可见性、weight、confidence、access count 选择。
- RRF 融合：避免只靠一个分数。
- 注入记录：被注入后更新 last injected 和 access count，用于后续质量评分。
- 失败降级：向量检索失败时回退基础 memory context。

### 4. 怎么做记忆压缩？

分两类：

- 对话上下文压缩：当 token 超阈值时，把旧消息摘要成 system summary，保留最近消息。项目里的 `ContextCompactionService` 会保留最近若干条，把 overflow 消息生成中文摘要，并标记旧消息不再用于上下文。
- Memory 抽象压缩：把多条 atomic memory 归并成 concept/profile，减少注入数量，提高稳定性。

### 5. 怎么看 Claude Code 的 memory 设计？

优点是把 memory 作为工程机制而不是“随便把历史拼进 prompt”：它有 session memory、压缩、阈值触发、工具调用统计和上下文边界。缺点是如果没有用户级治理、冲突处理和评测，很容易产生陈旧记忆或错误偏好。

本项目在此基础上增加了用户级 memory item、可见性、反馈、冲突处理、向量召回和后台维护。

### 6. Memory benchmark 怎么设计？

建议指标：

- 记忆抽取准确率：该记的是否记了，不该记的是否没记。
- 冲突处理准确率：新旧偏好是否正确取舍。
- 检索 Recall@K：问到相关内容时能否召回。
- 注入有效性：注入 memory 后答案是否更好。
- 污染率：跨用户、跨会话、无关 memory 注入比例。
- 遗忘能力：用户删除/关闭 memory 后是否不再使用。

### 6A. 记忆机制完整怎么做？短期记忆和长期记忆怎么区分、存储和更新？

短期记忆服务当前任务，长期记忆服务跨会话个性化和稳定事实。

短期记忆：

- 来源：最近消息、当前任务 plan、工具结果、附件临时摘要、session summary。
- 存储：session message、job trace、短期 cache 或 summary。
- 生命周期：跟随会话或任务结束，过期后只保留摘要或按需检索。
- 作用：维持多轮连续性，避免每轮重新解释上下文。

长期记忆：

- 来源：用户明确表达的偏好、稳定事实、长期项目背景、可复用经验。
- 存储：memory item 表 + 向量索引，带 user/session、category、confidence、source、status、timestamps。
- 生命周期：可编辑、可删除、可归档，需要治理和冲突处理。
- 作用：跨会话个性化和减少重复询问。

记忆更新策略一般不是“每轮都直接写长期 memory”。更稳的是 candidate 流程：

1. 从对话或工具结果抽取 memory candidate。
2. 判断是否值得记：稳定性、隐私性、是否用户明确、是否已存在。
3. 去重和冲突检测：hash、语义相似、同 namespace/category 冲突。
4. 置信度决策：高置信自动写入，低置信 pending confirm，高风险不写。
5. 注入后反馈：记录 access count、last injected、用户删除/纠错。

面试里可以强调：长期 memory 一定要有用户控制和可追溯来源，否则错误记忆会比没有记忆更危险。

## 六、工具调用、容错、流式输出

### 1. Agent 使用多少外部工具？如何保障调用链容错和超时？

项目里的工具包括文件、bash、web、MCP、skill、tasks、artifact 等。容错设计包括：

- `context.Context` 贯穿工具执行，支持取消和超时。
- Bash 和 sandbox bash 有 timeout、输出大小限制。
- LLM governance 有重试、熔断、quota、timeout。
- job 有状态机、事件流和取消。
- skill sandbox 可用 Docker 隔离，限制网络、内存、CPU、PIDs、tmpfs。

### 2. 工具调用失败后的 feedback 策略怎么设计？

可以回答：

> 工具失败不会直接吞掉，而是作为 tool result 或 job event 进入 trace。模型可以基于错误信息选择重试、换工具、降级或向用户说明。系统侧会统计 tool error rate，评测侧把工具错误作为 failed finding。对可重试错误如 timeout、临时网络失败，可以指数退避；对权限拒绝、参数错误，不盲目重试。

### 3. 需要支持 Streaming 输出，但延迟超标，怎么折中？

分开看 TTFT 和完整耗时：

- 首 token 慢：减少首轮检索/工具阻塞，先发状态事件或轻量回答。
- 完整耗时慢：工具并行、异步任务化、长任务走 job event stream。
- 检索慢：设置检索超时、fallback、缓存常用 embedding。
- 模型慢：小模型路由、降低输出长度、必要时后台继续。

本项目支持 SSE chat/job stream，也记录 TTFT 和 latency；长任务用 job 事件流比同步 HTTP 更稳。

## 七、模型微调、LoRA、DPO、vLLM 与推理

### 1. LoRA 是什么？用来做什么？

LoRA 是参数高效微调方法：冻结大模型原始权重，只在部分线性层旁边加低秩矩阵，训练少量参数来适配特定任务。它适合指令风格、领域格式、工具调用习惯、特定输出结构等场景。

结合项目要诚实说：当前仓库核心不是训练 LoRA，而是 Agent 工程化、检索、memory、tool/runtime。如果简历写 LoRA，需要明确这是另一个训练实验，和本项目的 online runtime 解耦。

### 2. LoRA 的优势和短板？

优势：训练成本低、显存占用小、可插拔、便于多任务 adapter 管理。短板：知识注入能力有限，容易过拟合小数据；对复杂推理提升有限；多 adapter 合并和线上版本管理会增加复杂度。

### 3. QLoRA 是什么？

QLoRA 是在量化基座模型上做 LoRA 微调，通常用 4-bit 量化降低显存，再训练 adapter。它更适合资源受限场景，但训练稳定性、量化误差和推理部署链路要额外验证。

### 4. 什么时候做 SFT，什么时候适合 RL/DPO？

- SFT：模型不会按业务格式输出、工具调用格式不稳定、领域话术需要对齐时。
- DPO/RL：已经有多个候选回答和明确偏好，想优化“哪个更好”，例如更少幻觉、更好遵循约束、更会拒答。

Agent 任务里，我会先做 RAG、prompt、工具 schema 和评测闭环。只有当 badcase 明确来自模型行为模式，而不是检索/工具/上下文问题时，才考虑 SFT 或 DPO。

### 5. DPO 相比 SFT 有哪些优劣？偏好对怎么构造？

DPO 优点是直接利用偏好对优化回答偏好，不需要 RLHF 的复杂 reward model；缺点是偏好对质量决定上限，且容易学到表面风格偏好。偏好对可以来自线上 badcase 修正、人工标注、多模型候选对比，也可以自动生成后人工抽检。

### 6. vLLM 为什么快？KV-cache 如何加速？

KV-cache 缓存 transformer 已经计算过的 key/value，生成下一个 token 时不用重复计算历史 token。vLLM 的优势在于 PagedAttention 和 continuous batching，能更高效管理 KV-cache 内存，把不同请求动态合批，提高吞吐。

如果问到项目：本项目默认 provider 是 Vertex AI，并有多 provider 抽象和 LLM governance；仓库没有自部署 vLLM 的核心代码，所以可以讲原理和如果自部署会怎么接入，而不要说项目里已经深度优化了 vLLM。

### 7. KV cache 碎片化怎么解决？Continuous batching 原理是什么？

KV cache 碎片化来自不同请求长度和生命周期不同，导致显存块利用率低。PagedAttention 把 KV cache 切成固定大小 block，用类似虚拟内存的方式管理，减少连续大块分配需求。

Continuous batching 是推理服务在每个 decoding step 动态加入新请求、移除完成请求，而不是等一整个 batch 全部结束，从而提高 GPU 利用率并降低排队浪费。

## 八、后端工程与生产化

### 1. 多用户并发下 memory 怎么隔离？如何保证线程安全？

项目层面用 `user_id`、`session_id` 做数据隔离，Memory、Search、Job、Artifact 都带用户维度。代码层面 Go 的共享运行态使用 mutex、context 和 store 接口隔离；数据库层面 SQL service 按 user/session 查询；前端 API 使用认证 token 和 CSRF。

如果被问 LangChain memory 默认机制，可以说：默认内存对象不适合直接多用户共享，必须按 user/session 创建独立 memory store，不能把全局 ConversationBufferMemory 复用给所有请求。

### 2. 多轮对话上下文状态管理如何做？高并发怎么保证一致性？

项目里 session 是核心状态单元，消息写入走 message service/store；长任务走 job queue 和 job event；上下文加载有 max messages、max tokens、summary 等策略。高并发下要保证：

- 写入消息有顺序号或事务边界。
- 同一 session 的并发写要串行化或做冲突控制。
- 长任务状态变更有 terminal state 保护。
- SSE/事件流支持重连和 replay。

### 3. 冷启动怎么解决？

分用户冷启动和系统冷启动：

- 用户冷启动：没有长期 memory 时，优先用当前会话、附件、显式 profile、浏览器 memory、默认 prompt。
- 检索冷启动：没有向量索引时走 SQL/全文 fallback；索引异步补齐。
- Skill 冷启动：预加载/缓存 skill registry，Docker sandbox 可做 warm pool。
- 服务冷启动：readiness 检查 Postgres、Redis、对象存储、LLM provider。

### 4. 如果设计一个大型 Agent 系统，整体怎么分层？

可以按本项目回答：

1. 接入层：Web/API、认证、CSRF、rate limit。
2. 会话层：session、message、attachment、artifact。
3. Agent runtime 层：query loop、tool registry、permission、provider、compaction。
4. 能力层：Skills、MCP、memory、search、tasks、sub-agent/team。
5. 治理层：评测、审计、风险、成本、quota、LLM governance。
6. 基础设施层：Postgres、Redis、S3/R2、Qdrant/Elastic、Kafka、Prometheus。

### 5. Agent 哪些模块最容易在真实业务中出问题？如何监控定位？

最容易出问题的是检索、工具、记忆、权限、模型 provider 和流式链路。监控要看：

- 检索：召回为空率、Qdrant/Elastic 延迟、索引失败。
- 工具：错误率、超时率、sandbox 启动时间。
- 模型：TTFT、P95 latency、失败率、成本、quota。
- Memory：错误注入、冲突、过期、用户删除后仍使用。
- 流式：断连、重连、事件 replay。

本项目有 Prometheus metrics、evaluation trace、LLM usage、risk events、admin console，可以支撑定位。

### 5A. 系统设计题：1000 条数据处理怎么做，扩大到百万级怎么处理？

先问清处理类型：是 CPU 计算、外部 API 调用、数据库写入、embedding 生成，还是 LLM 批处理。不同瓶颈决定架构。

1000 条数据可以用简单批处理：

1. 参数校验和去重。
2. 分批读取，例如每批 100 条。
3. worker pool 并发处理，控制并发数。
4. 结果批量写入，失败记录 error table。
5. 任务状态可查询，支持重试和取消。

百万级时要升级成分布式流水线：

- 数据分片：按 tenant、时间、ID range 或 hash 分片。
- 队列解耦：用 Redis Stream/Kafka/SQS，把生产和消费拆开。
- 幂等处理：每条数据有 task id / idempotency key，重复消费不重复写。
- 批量优化：批量读写、批量 embedding、批量 upsert 向量库。
- 背压限流：队列水位、worker 并发、外部 API quota、LLM token budget。
- Checkpoint：记录每个分片 offset、状态、失败次数和最后处理时间。
- 失败隔离：可重试错误延迟重试，不可重试进入死信队列。
- 观测告警：吞吐、积压、P95、失败率、重试率、成本。

如果是 RAG 索引构建，百万级重点是离线化和增量化：文档解析、chunk、embedding、全文索引、向量 upsert 分阶段执行；新数据增量索引，历史数据后台重建；查询线上链路和索引构建链路隔离，避免重建拖垮用户请求。

## 九、GraphRAG、Embedding、调优类追问

### 1. GraphRAG 怎么做？什么场景适合？

GraphRAG 是在普通 chunk 检索之外，抽取实体、关系和社区摘要，用图结构帮助回答跨文档、多跳关系问题。适合组织知识、供应链、商品关系、政策条款、用户行为链路等关系密集场景。

本项目当前更偏消息/memory 的 hybrid search，不是完整 GraphRAG。如果要扩展，可以把 memory item、session、artifact、skill execution 建成图节点，把引用、冲突、父子摘要、同会话关系作为边，再结合向量召回和图扩展。

### 2. Embedding 模型怎么选？

看四点：

- 语种：中文/英文/多语。
- 领域：代码、客服、商品、通用文本。
- 成本和延迟。
- 与向量维度、Qdrant collection、索引重建成本的兼容性。

项目里支持 OpenAI/Vertex embedding，Vertex 默认可用 `gemini-embedding-2`，并区分 query task type 和 document indexing task type。

### 3. Qdrant 怎么调优？

可以从这些方向答：

- collection 的向量维度和 distance 与 embedding 模型一致。
- payload filter 一定要带 `user_id`、`status`，减少无效搜索。
- 调整 limit、score threshold、HNSW 参数。
- 批量 upsert，附件文本分块控制 chunk size 和 overlap。
- 对 memory 和 message 分 collection，便于隔离生命周期。

本项目已经把 message 默认 collection 设为 `agent_messages`，memory 默认 collection 设为 `agent_memories`。

### 4. 为什么这个方案适合业务？换场景还成立吗？

适合 Web Agent 的原因是：用户问题既有语义模糊，也有精确检索；同时有多用户隔离、长对话记忆、附件、工具调用和线上评测需求。换到强结构化业务，比如订单状态查询，应该优先查数据库/API；换到知识库问答，则可以加 rerank、citation 和 GraphRAG。

## 十、基础与八股问题

### 1. Docker 和 VM 的区别？Docker 怎么隔离不同版本依赖？

VM 虚拟化的是硬件和完整 OS，隔离强但启动慢、资源重。Docker 共享宿主机内核，通过 namespace 隔离进程/网络/文件系统，通过 cgroup 限制 CPU/内存，通过 image/layer 固化依赖。

项目里 skill sandbox 就利用 Docker 隔离：限制 network、memory、CPU、PIDs、tmpfs，并把 skill root 只读挂载，适合执行不完全可信的技能脚本。

### 2. Go GMP / N:M 线程模型怎么理解？

G 是 goroutine，M 是 OS thread，P 是 processor/调度上下文。Go 通过 P 把大量 goroutine 调度到有限 OS thread 上，实现 M:N 并发。适合本项目这种大量 IO、HTTP、SSE、worker、tool execution 的场景，但共享状态仍要 mutex/channel/context 控制。

### 3. MySQL / Redis 可以结合项目怎么讲？

本项目默认是 Postgres，但关系型数据库关注点类似：事务、索引、唯一约束、分页、慢查询、连接池。Redis 在项目里用于 cache、rate limit、job queue/stream、processed lock 等，适合高频状态和队列，不适合作为唯一事实源。

### 4. Java / Python 基础被问到怎么办？

按真实掌握回答，不要硬套项目。可以把项目主语言拉回 Go/TypeScript：

- Python tuple/list：tuple 不可变，list 可变；不可变对象适合作 dict key 或表达固定结构。
- Python GC：引用计数为主，循环引用由 generational GC 处理。
- Java：如果不熟，承认主项目不是 Java，转而强调后端工程共性，如并发、连接池、事务、缓存一致性。

### 5. 分布式计算了解吗？

可以从项目生产化角度说：

- API 和 worker 可拆分。
- Redis stream/job queue 处理异步任务。
- Postgres 是事实源，Redis 是队列和缓存。
- 对象存储保存附件和 artifact。
- 搜索/向量服务可作为独立基础设施扩展。
- 关键是幂等、重试、可观测和状态一致性。

### 6. 怎么解决过拟合？

训练层面：数据去重、验证集、早停、正则、dropout、降低 LoRA rank、控制训练轮数。Agent 应用层面也有“过拟合”：prompt 只适配少数 badcase、评测集太窄。解决方式是按场景分桶、保留线上回放集，避免为了某个样本破坏整体行为。

## 十一、手撕题准备

常见题：

- 最大正方形：动态规划，`dp[i][j] = min(上, 左, 左上) + 1`。
- 归并排序：分治 + merge。
- 岛屿数量：DFS/BFS/并查集。
- 接雨水：双指针或单调栈。
- 最长子串，每个字符出现次数都 >= k：分治。若某字符总次数小于 k，它不可能出现在合法子串中，用它切分后递归。
- 硬币概率：先定义样本空间，再用条件概率/期望，不要急着套公式。

## 十二、面试回答策略

### 1. 面试官反复问“为什么这么设计”时怎么答？

用这个结构：

1. 先说业务约束：多用户、长任务、工具风险、成本、延迟、可观测。
2. 再说方案选择：为什么用 hybrid search、memory item、SSE、job queue、sandbox。
3. 说替代方案：为什么不用纯向量、纯 prompt、全自动 multi-agent。
4. 说边界：在哪些场景不成立，未来怎么改。

### 2. 怎么讲清楚“我在项目里的角色”？

建议明确到模块：

> 我主要负责 Agent runtime 和产品后端的衔接，包括会话消息、流式响应、工具/Skill 调用、Memory 抽取和检索、评测 trace、LLM governance、部署配置等。不是单纯调 API，而是把模型能力接到可运营的 Web 产品里。

如果某块只是了解，不要说主导。比如 LoRA/vLLM 可以说“了解原理，项目当前没有作为主链路落地”。

### 3. 如果让你重做一遍，会怎么改？

可以答：

- 更早建立离线 eval 集和线上 trace 回放，而不是后补评测。
- 检索从一开始就保留 gold evidence 标注，方便衡量 Recall@K。
- Memory 增加更强的用户确认和可解释 UI。
- 工具调用加入更细粒度的 retry policy 和 circuit breaker。
- 对 Agent workflow 做更清晰的任务边界，简单任务不引入 Multi-Agent。

## 十三、图片补充：AI Agent / AI 应用二面高频追问

这一组题来自图片里的字节飞书、腾讯 AI 应用开发、Agent 二面和 Redis/Transformer 八股。前面已覆盖过的问题这里不重复铺开，只补充更适合面试追问的回答口径。

### 1. 说说你对 LangGraph 的理解。

LangGraph 可以理解为“面向 Agent 工作流的有状态图编排框架”。它把一个 Agent 流程拆成 `State`、`Node`、`Edge` 和 `Checkpoint`：

- `State`：保存当前任务上下文、历史消息、工具结果、路由状态等。
- `Node`：一个可执行步骤，例如意图识别、检索、工具调用、人工审核、生成回答。
- `Edge`：节点间跳转逻辑，可以是固定边，也可以是条件边。
- `Checkpointer`：把每一步状态持久化，支持中断恢复、重放和人工介入。

和普通 LangChain 链式调用相比，LangGraph 更适合多轮、有分支、需要恢复的 Agent 流程；和手写状态机相比，它降低了状态传递、条件路由和 checkpoint 的样板代码。

### 2. LangGraph 和 LangChain 有什么区别？

LangChain 更偏组件和链：模型、prompt、retriever、tool、chain，适合快速搭 RAG 或简单工具链。LangGraph 更偏有状态编排：节点、边、状态、循环、条件分支和 checkpoint，适合复杂 Agent。

可以这样总结：LangChain 解决“有哪些 LLM 应用组件”，LangGraph 解决“这些组件如何按状态机稳定流转”。

### 3. LangGraph 里的 State、Node、Edge 分别解决什么问题？

- `State` 解决上下文共享和状态恢复问题，不需要每个节点自己拼参数。
- `Node` 解决职责拆分问题，把检索、生成、工具调用、审核等步骤隔离。
- `Edge` 解决流程控制问题，比如检索失败走 query rewrite，工具高风险走人工审批。

面试时可以补一句：State 不应该无限膨胀，最好只存必要状态，长文本和大附件放外部存储，只在 State 里放 ID 和摘要。

### 4. 为什么选择 LangGraph，而不是 AutoGen、CrewAI 或手写状态机？

选择依据不是“哪个更火”，而是任务形态：

- LangGraph：适合可控流程、有状态、多分支、可恢复的生产 Agent。
- AutoGen/CrewAI：适合多角色协作探索，但生产可控性和成本治理要额外补。
- 手写状态机：性能和定制最强，但容易把路由、状态恢复、观测、重试都写散。

结合本项目，我更偏“核心链路 workflow 化，复杂步骤交给 Agent”。如果用 LangGraph，我会把认证、权限、任务状态、检索、工具调用、审核这些明确环节做成节点，而不是让模型自由决定所有流程。

### 5. 框架能力不满足时怎么扩展？

常见扩展点：

- 自定义 node：接入内部检索、风控、数据库、审批系统。
- 自定义 state schema：保存业务 ID、权限范围、工具结果、失败原因。
- 自定义 checkpointer：落 Postgres/Redis，而不是只用内存。
- 自定义 tool wrapper：统一 timeout、重试、审计、权限和参数校验。
- 自定义 callback/trace：把每一步延迟、token、工具错误写入观测系统。

本项目实际上就是类似思路：runtime 不直接依赖某个框架，而是自己抽象了 session、job、tool、skill、memory、evaluation、LLM governance 等边界。

### 6. 项目中多智能体怎么调度？有哪些智能体和职责？

可以按“角色 + 调度 + 终止”回答：

> 我不会默认所有任务都拆多 Agent。项目里适合拆的场景是代码任务、复杂分析和质量验证。常见角色包括 planner/architect 负责拆解方案，executor 负责实现，verifier/test-engineer 负责验证，reviewer 负责风险和质量复核。  
> 调度上由主流程分配子任务，子 Agent 输入必须有明确目标、上下文边界和交付物；输出回到主 Agent 汇总。终止条件包括任务完成、验证通过、达到迭代上限、遇到不可恢复错误或需要人工决策。

避免死循环的关键是：每个 Agent 不直接互相无限对话，而是通过任务状态、结果 artifact、测试证据和主控调度收敛。

### 7. 基于多智能体的智能问答平台有哪些功能？你负责哪些？

可以结合本项目回答：

功能层面：

- 用户登录、会话管理、消息流式输出、附件/Artifact。
- RAG 检索、memory 注入、上下文压缩。
- Skill/MCP 工具调用、长任务 job、取消和事件流。
- 管理后台、审计、风控、成本和评测。

个人职责可以说：

> 我主要负责 Agent runtime 和产品后端衔接，包括会话消息、SSE/job event、工具/Skill 调用、Memory 抽取与检索、评测 trace、LLM governance、部署配置等。重点是把模型能力接到可控、可观测、可恢复的 Web 产品里。

### 8. 怎么设计 Tool Calling？

设计 Tool Calling 时不能只把工具描述丢给模型，至少要有五层：

1. Schema：输入输出用 JSON schema 或强类型结构定义，必填字段、枚举、范围要明确。
2. 权限：工具按读/写/执行/高风险分级，危险工具需要审批或 sandbox。
3. 执行：服务端统一分发，设置 timeout、重试、最大输出、取消。
4. 错误：参数错误、权限拒绝、工具异常、外部服务失败要区分处理。
5. 观测：记录 tool name、参数摘要、耗时、状态、错误和 trace ID。

本项目里工具不是直接暴露给模型执行，而是经过 runtime/tool registry、permission、sandbox、job event 和 evaluation trace。

### 9. Tool 的输入输出 schema 怎么定义？

输入 schema 要面向模型可理解，也要面向服务端可校验：

- 字段名语义清晰，避免 `data`、`params` 这种大包字段。
- required 字段尽量少，但关键字段必须明确。
- enum、数字范围、路径范围、URL 白名单要写进 schema 或后端校验。
- 输出要稳定，至少包含 `status`、`data`、`error`、`metadata`。

面试可以强调：schema 是模型和业务系统的契约，不是 prompt 文案。Prompt 可以解释怎么用，schema 和后端校验负责兜底。

### 10. 工具调用失败、参数错误怎么兜底？

先分类：

- 参数错误：把校验错误反馈给模型，让它 repair 一次；连续失败则向用户澄清。
- 权限错误：不要重试，提示需要授权或人工审批。
- 临时错误：网络、超时、限流可以指数退避重试。
- 业务错误：比如订单不存在，返回可解释结果，不让模型编。
- 高风险错误：写操作失败要保持幂等，避免重复扣费、重复删除。

本项目里可以用 job status、tool result、evaluation trace 记录失败，后续定位 badcase 时能知道是 prompt、模型、工具还是外部服务问题。

### 11. 如何防止模型调用危险工具？

核心原则：模型只能“申请调用”，不能绕过服务端执行控制。

具体做法：

- 工具分级：读工具、写工具、执行工具、高风险工具。
- 权限策略：按用户、会话、skill、环境限制 allowed tools。
- 参数校验：路径、域名、命令、SQL、文件类型都要做白名单或沙箱。
- 人工审批：删除、支付、发消息、发布、权限变更等动作需要确认。
- 审计和回滚：记录调用链，必要时支持撤销或补偿。

项目里的 Docker skill sandbox、permission mode、risk review、audit log 都是这类治理能力。

### 12. 如何判断是 Prompt 问题还是模型能力问题？

用消融实验定位：

- 固定模型版本、固定输入，只改 prompt，如果改善明显，多半是 prompt 问题。
- 固定 prompt，换模型或模型版本，如果只有高能力模型能稳定解决，多半是模型能力问题。
- 给足 gold evidence 后仍答错，可能是推理或指令遵循问题。
- 人工把工具结果改成标准格式后正常，可能是工具输出或上下文组织问题。
- 多次采样波动大，可能是解码参数或 prompt 约束不足。

线上要结合 trace：看检索结果、prompt、模型版本、参数、工具结果、输出和用户反馈，而不是只看最终答案。

### 13. 项目中有没有遇到模型不听指令？怎么解决？

可以用一个通用 badcase 回答：

> 遇到过模型在上下文不足时编造答案，或者工具参数缺字段还强行调用。定位时我会回放 trace，检查检索 evidence 是否命中、工具 schema 是否清晰、prompt 是否有冲突指令、模型参数是否过于发散。  
> 解决上优先用流程约束和后处理，而不是只加 prompt：比如强制 JSON schema、参数校验失败后 repair、无 evidence 时拒答、高风险动作走审批、生成后做 groundedness/format check。解决后会把 badcase 加入 golden set 或 trace 回放集。

反思结果污染上下文的问题，要避免把错误反思直接写入长期 memory。反思只用于当前任务或评测记录，进入长期 memory 前要有置信度、来源和用户确认。

### 14. Prompt 工程的边界是什么？为什么不能把一切都交给 Prompt？

Prompt 适合表达目标、约束、风格和少量示例，但不适合承担安全边界、权限控制、事务一致性和格式强校验。

不能全交给 Prompt 的原因：

- 模型可能忽略或误解指令。
- Prompt 越长越容易互相冲突。
- 线上业务需要可审计、可测试、可回滚。
- 安全动作必须由服务端权限和流程控制。
- 结构化输出必须由 schema/parser 校验。

面试中可以说：Prompt 是软约束，工程系统必须提供硬约束。

### 15. 结构化输出有哪些常见做法？优缺点是什么？

- Prompt + 示例：实现最快，但稳定性最差。
- JSON mode / structured output：稳定性较好，但复杂 schema 和长文本仍需校验。
- Function calling / tool call：适合动作和参数抽取，天然有 schema。
- 生成后 parser + repair：工程兜底能力强，但多一次模型调用会增加延迟。
- 微调：长期稳定任务有效，但成本高、迭代慢。

线上推荐组合：structured output 或 function calling + schema 校验 + repair retry + 失败降级。

### 16. Golden set 应该怎么构建？

Golden set 不是随便攒几条问答，而是要覆盖真实业务风险：

- 来源：线上高频 query、badcase、人工构造边界样本、历史事故、核心业务流程。
- 分层：检索、工具调用、memory、长上下文、安全拒答、结构化输出、延迟敏感场景。
- 标注：标准答案、必须命中的 evidence、允许的表达差异、禁止内容。
- 版本：记录数据版本、prompt 版本、模型版本和评测脚本版本。
- 维护：线上新 badcase 定期回灌，过期业务规则及时下线。

本项目可以把 runtime trace、evaluation result、risk event、tool error 和用户反馈作为 golden set 的候选来源。

### 17. 线上效果波动，怎么判断是 Prompt 问题还是模型版本问题？

先看变更时间线：

1. Prompt、检索配置、rerank、tool schema、模型版本、解码参数是否有发布。
2. 用同一批 golden set 回放旧模型/新模型、旧 prompt/新 prompt。
3. 分桶看波动：只影响结构化输出，多半是 prompt/schema；只影响推理题，可能是模型；只影响知识问答，可能是检索或索引。
4. 看线上指标：TTFT、错误率、空输出率、tool error、召回率、用户反馈。

不要直接凭感觉改 prompt，要用 A/B 或离线回放把变量拆开。

### 18. 请描述一个完整的 RAG 流水线。

完整流水线分离线和在线：

- 离线：文档解析、清洗、切 chunk、抽 metadata、embedding、写入向量库/全文索引、索引质检。
- 在线：query 理解、query rewrite、权限过滤、多路召回、rerank、上下文拼装、LLM 生成、引用/格式校验、日志评测。
- 反馈：用户反馈、badcase 归因、golden set 回灌、索引和 prompt 迭代。

本项目的消息、附件和 memory 都能进入检索链路，最后只把用户可见、状态 active、相关性足够的内容注入上下文。

### 19. RAG 中 chunk 应该怎么切？

chunk 策略取决于文档结构和问答粒度：

- 优先按语义结构切：标题、段落、列表、代码块、表格。
- 保持 chunk 自包含：带上标题路径、章节名、必要上下文。
- 设置 overlap：避免答案跨 chunk 被切断。
- 控制长度：太短缺上下文，太长影响召回和 prompt 成本。
- 特殊内容特殊处理：代码按函数/类切，表格按行组或转摘要，FAQ 按问答对切。

经验上不要只按固定 token 硬切，最好“结构优先，token 兜底”。

### 20. RAG 的 metadata 应该有哪些字段？

常见字段：

- 基础：`doc_id`、`chunk_id`、`source`、`title`、`url`、`created_at`、`updated_at`。
- 权限：`user_id`、`org_id`、`visibility`、`acl`、`status`。
- 结构：`section_path`、`page`、`heading`、`chunk_index`、`parent_id`。
- 检索：`embedding_model`、`embedding_version`、`language`、`content_type`。
- 业务：`product_id`、`session_id`、`memory_id`、`tags`、`category`。

本项目里的 message/memory vector payload 会保存 `user_id`、`session_id`、`message_id`、`role`、`content`、`status` 等，核心就是为了过滤、回查和审计。

### 21. RAG 为什么要加 rerank？

召回阶段追求“别漏”，通常 topK 会带入一些噪声；rerank 阶段追求“排准”，把最相关、最能支撑答案的 evidence 放到前面。

加 rerank 的收益：

- 多路召回结果分数尺度不同，rerank 可统一排序。
- 向量召回语义相关但不一定回答问题，rerank 更关注 query-document 匹配。
- 可以减少最终注入 chunk 数，降低 LLM 成本和幻觉。

代价是延迟和成本增加，所以可以只对召回 top 50 做 rerank，再取 top 5 到 10 注入。

项目改造状态：

- [x] `MessageSearchService` 已支持轻量本地 rerank：对 RRF/语义召回候选按 query 词项匹配、原始融合分和时间新鲜度重排，避免多路召回分数尺度不一致时把弱相关内容排在前面。
- [x] rerank 候选窗口可通过 `AGENT_API_MESSAGE_SEARCH_RERANK_CANDIDATE_LIMIT` 控制，开关为 `AGENT_API_MESSAGE_SEARCH_RERANK_ENABLED`。

### 22. Recall@K 和 Precision@K 怎么取舍？

- `Recall@K`：标准 evidence 是否出现在前 K 个结果里，衡量“有没有找回来”。
- `Precision@K`：前 K 个结果里有多少是真相关，衡量“噪声多不多”。

RAG 通常先保证 Recall@K，因为没召回正确 evidence，生成阶段基本无解；然后通过 rerank、过滤和上下文压缩提高 Precision@K。面试可以说：召回阶段宁可多一点，注入阶段必须精。

### 23. Top-K 怎么动态调整？

可以按 query 和结果质量动态调：

- query 很明确、top 分数断崖式领先：小 topK。
- query 模糊、分数接近、多跳问题：扩大 topK 或触发 query rewrite。
- rerank 后高相关结果少：扩大召回窗口。
- 延迟敏感场景：降低 topK，必要时走缓存或 FAQ。

项目里可以把 topK 和 timeout 做成配置，并用评测观察 Recall、latency、cost 的变化。

项目改造状态：

- [x] 消息 hybrid/semantic 搜索已支持动态召回窗口：`AGENT_API_MESSAGE_SEARCH_DYNAMIC_TOPK_ENABLED` 打开后，根据 query 模糊程度、是否包含“上次/之前/那个”等多轮指代词，在 `MIN_RECALL_WINDOW` 和 `MAX_RECALL_WINDOW` 之间调整候选数。
- [x] 最终返回给前端的 `limit/offset` 不变，动态 TopK 只影响后端候选召回和 rerank 窗口，不会改变 API 分页语义。

### 24. 如果 BM25 已经很好，向量检索还有必要吗？

看场景。如果用户问题和文档表达高度一致，BM25 可能足够，而且更快、更可解释。但向量检索仍有价值：

- 用户自然语言改写多。
- 文档同义表达多。
- 需要找“意思相近”而不是“字面相同”的内容。
- Memory、会话摘要、经验类知识往往更适合语义召回。

实际方案可以不是二选一：BM25 做主召回，向量做补召回，用 RRF/rerank 融合。若评测证明向量召回贡献很低，可以降权或只在特定 query 触发。

### 25. Agentic RAG 和传统 RAG 有什么区别？

传统 RAG 通常是固定链路：query -> retrieve -> generate。Agentic RAG 会让 Agent 根据中间结果自主决定是否改写 query、换检索源、调用工具、多轮检索或拒答。

Query Rewrite 本身不一定算 Agentic。如果 rewrite 是固定规则或固定链路，它仍是传统 RAG 的增强；如果系统会基于检索失败、证据冲突、任务目标动态选择下一步，才更接近 Agentic RAG。

Agentic RAG 的优势是灵活，缺点是成本、延迟和不稳定性更高，所以需要最大轮数、预算、失败分支和 trace。

### 26. 多轮检索如何设计？检索失败后 Agent 怎么调整策略？

多轮检索可以这样设计：

1. 首轮用原 query 做 hybrid search。
2. 如果无结果或分数低，抽取实体/关键词重写 query。
3. 如果证据冲突，按时间、来源可信度或业务优先级二次检索。
4. 如果缺实时数据，调用 API/tool，而不是继续搜静态知识库。
5. 达到轮数或预算上限仍无证据，则拒答或请求用户补充。

检索失败后的策略不是“无限搜索”，而是有条件、有预算地 rewrite、扩大范围、换索引、查工具或澄清。

项目改造状态：

- [x] message hybrid search 已增加确定性 query rewrite：保留原 query，同时生成去口语化和关键词版本作为补召回查询。
- [x] 已增加低置信二轮检索：首轮候选少于目标数量或 top score 低于 `AGENT_API_MESSAGE_SEARCH_LOW_CONFIDENCE_SCORE` 时，才用 rewrite 版本补召回，再用 RRF 合并，避免无限搜索。
- [x] 相关配置包括 `AGENT_API_MESSAGE_SEARCH_QUERY_REWRITE_ENABLED`、`AGENT_API_MESSAGE_SEARCH_MULTI_TURN_ENABLED`、`AGENT_API_MESSAGE_SEARCH_LOW_CONFIDENCE_SCORE`。

### 27. LangGraph 里的 Checkpointer 和长期记忆有什么区别？

Checkpointer 保存的是流程运行状态：当前节点、State、消息、中间工具结果，用于中断恢复、重放和调试。长期记忆保存的是跨任务可复用的用户偏好、事实、项目规则和经验，用于后续上下文增强。

简单说：checkpointer 是“这次任务执行到哪了”，长期记忆是“以后还值得记住什么”。Checkpointer 不应该当知识库，长期记忆也不应该保存每一步临时状态。

### 28. Claude Code 的 Memory 机制你了解吗？为什么要分层记忆？

分层是为了避免所有信息混在一个上下文里：

- 项目级规则：代码风格、构建命令、架构约束。
- 用户偏好：回答风格、常用约定。
- 会话上下文：当前任务目标、已完成步骤、未解决问题。
- 短期摘要：长对话压缩后的当前状态。

隔离原则是：项目规则不等于用户偏好，会话临时结论不应自动变成长期事实。上下文过长时要摘要、检索和裁剪，而不是无限追加。

### 29. 怎么看扣子这类低代码 Agent 平台？和 LangGraph 怎么选？

低代码平台优势：

### 33. Agent 链路耗时很长，如何定位性能瓶颈？

先把链路拆成指标：

- 意图识别耗时。
- 检索耗时：embedding、向量库、全文检索、rerank。
- LLM 耗时：TTFT、总生成时间、输入/输出 token。
- 工具耗时：外部 API、sandbox 启动、文件处理。
- 队列耗时：job 排队、worker 执行、事件发送。

### 34. 意图识别 / 任务路由应该怎么做？

可以做分层路由：

1. 规则路由：明显命令、URL、文件、FAQ、权限动作，先用规则。
2. 轻量分类：规则不够时用小模型或分类器。
3. 语义路由：复杂意图再用 LLM 判断。
4. 人工/澄清：高风险或歧义很大的意图先问清楚。

规则适合高精度、低成本、稳定模式；语义分类适合表达多变、意图模糊的问题。FAQ 和 Tool 同时命中时，可以先回答事实，再询问是否执行动作；如果用户明确要求执行，则走工具链。

### 35. 如何避免错误路由把后续链路全部带偏？

做法是：

- 路由结果带置信度，低置信度不直接进入高风险链路。
- 保留 fallback，例如 FAQ -> RAG -> tool -> 澄清。
- 每个关键节点重新校验前置条件。
- 工具执行前做参数和权限二次确认。
- trace 记录路由原因，badcase 可回放。

不要让第一次分类成为不可纠错的单点决策。

### 36. Redis 常见数据结构有哪些？分别适合什么场景？

- String：缓存 JSON、token、计数器、分布式锁值。
- Hash：用户 profile、会话状态的小字段集合。
- List：简单队列、最近消息列表，但复杂消费确认不如 Stream。
- Set：去重集合、权限集合、标签集合。
- ZSet：排行榜、按时间排序任务、延迟队列。
- Stream：消息流、任务队列、事件日志，支持消费组和 ACK。
- Bitmap/HyperLogLog：签到、UV 估算等统计场景。

Agent 系统里的会话状态可以用 Hash 或 String JSON 加 TTL；事件流和异步任务更适合 Stream；长期事实源仍应放 Postgres。

### 37. Redis 的过期删除策略和内存淘汰策略是什么？

Redis key 过期后不一定立刻删除。删除策略主要是：

- 惰性删除：访问 key 时发现过期再删。
- 定期删除：后台周期性抽样扫描过期 key。

内存达到 `maxmemory` 后才触发淘汰策略，常见 `maxmemory-policy`：

- `noeviction`：不淘汰，写入报错。
- `allkeys-lru` / `volatile-lru`：按 LRU 淘汰所有 key 或只淘汰带 TTL 的 key。
- `allkeys-lfu` / `volatile-lfu`：按 LFU 淘汰。
- `allkeys-random` / `volatile-random`：随机淘汰。
- `volatile-ttl`：优先淘汰剩余 TTL 短的 key。

缓存会话状态时，如果 Redis 只放缓存，可以选 `allkeys-lru` 或 `allkeys-lfu`；如果混有不能丢的数据，最好只给缓存 key 设置 TTL，并用 `volatile-lru/volatile-ttl`，同时不要把 Redis 当唯一事实源。

### 38. 什么是缓存穿透、缓存击穿、缓存雪崩？

- 缓存穿透：查询不存在的数据，每次都打到 DB。解决：参数校验、缓存空值、布隆过滤器。
- 缓存击穿：热点 key 过期，大量请求同时打 DB。解决：互斥锁/singleflight、逻辑过期、后台刷新。
- 缓存雪崩：大量 key 同时过期或 Redis 故障。解决：随机 TTL、分批预热、多级缓存、限流降级。

缓存击穿加互斥锁是为了让同一时刻只有一个请求回源重建缓存；缓存雪崩设置随机过期是为了避免大量 key 同时失效。

### 38A. 分布式令牌桶、漏桶、滑动窗口限流怎么讲？Redis 用什么结构实现？

令牌桶适合“允许一定突发”的场景。桶里按固定速率生成 token，请求来时拿到 token 才能通过；桶容量决定最大突发量，生成速率决定长期平均 QPS。分布式实现时要把 token 数、上次补充时间放到 Redis，并用 Lua 脚本原子计算补充和扣减。

漏桶适合“平滑出流”的场景。请求进入桶，系统按固定速率处理；桶满则拒绝或排队。它能削峰，但对突发不友好，用户体验可能是排队延迟增加。

滑动窗口适合精确控制某个时间窗口内的请求数。结构体可以包含：

- `key`：限流对象，例如 user、tenant、IP、API。
- `windowSize`：窗口长度，例如 60 秒。
- `limit`：窗口内最大请求数。
- `now`：当前时间。
- `buckets` 或 `events`：分桶计数或请求时间戳集合。
- `currentCount`：当前窗口累计计数。
- `expireAt`：Redis 过期时间，避免冷 key 常驻。

滑动窗口和令牌桶的区别：

- 滑动窗口更关注“过去一段时间不能超过多少次”，精度高，但存储和计算开销更高。
- 令牌桶更关注“平均速率 + 允许突发”，实现简单，适合 API 网关和模型调用限流。
- 漏桶输出最平滑，但可能牺牲突发处理能力。

Redis 实现：

- 令牌桶：Hash/String 存 token 和 last_refill，用 Lua 原子更新。
- 滑动日志：ZSet 存请求时间戳，`ZREMRANGEBYSCORE` 删除窗口外记录，`ZCARD` 计数。
- 滑动分桶：Hash 存每个小时间片计数，降低 ZSet 内存。
- 固定窗口：String + `INCR` + `EXPIRE`，简单但边界突刺明显。

Agent 系统里一般会按租户、用户、模型、工具维度做多级限流：接入层保护 API，LLM governance 控制 token/成本，工具层保护敏感或昂贵接口。

### 39. 布隆过滤器有什么作用？

布隆过滤器用于快速判断“某个 key 一定不存在或可能存在”。它能防缓存穿透：如果布隆过滤器判断不存在，就不用查缓存和数据库。

缺点是假阳性：可能把不存在判断成可能存在，但不会把存在判断成不存在。适合商品 ID、用户 ID、文档 ID 这类集合过滤。

误判率由 bit 数组大小 `m`、元素数量 `n`、hash 函数个数 `k` 决定。经验上 `m` 越大误判率越低，`k` 不是越多越好，太多会让 bit 很快被打满。工程上要预估元素规模和目标误判率，按公式选择参数，并在数据增长后重建或扩容。

常见问题：

- 不能直接删除元素，除非用 Counting Bloom Filter。
- 数据增长超过预估后误判率会上升。
- 布隆过滤器只适合挡不存在的 key，不能替代权限校验和数据库事实源。

### 40. Redis Stream 和 Kafka 有什么区别？什么场景用哪个？

Redis Stream 是 Redis 内置的轻量消息流，支持持久化、消费组、Pending List 和 ACK；Kafka 是分布式日志系统，吞吐、持久化、分区扩展、跨服务数据管道能力更强。

Redis Stream 不等于绝对不丢消息。它依赖 Redis 持久化配置、复制和运维；如果没有 AOF/RDB 或发生极端故障，仍可能丢。Pending List 是消费组里已投递但未 ACK 的消息集合，可以用 `XACK` 确认，用 `XAUTOCLAIM` 或 `XCLAIM` 转移超时未处理消息。

选择：

- Redis Stream：轻量 job queue、实时状态事件、系统内部任务。
- Kafka：高吞吐、长期保留、多消费者数据管道、强审计日志。

本项目可以用 Redis stream 处理 job queue 和事件，用 Kafka 做更大规模的消息归档或跨系统流水。

### 41. Transformer 基础讲一下。

Self-Attention 的核心是 Q/K/V：

1. 输入向量分别线性变换成 Query、Key、Value。
2. 用 `QK^T / sqrt(d)` 计算 token 间相关性。
3. softmax 得到注意力权重。
4. 权重乘以 V 得到每个 token 融合上下文后的表示。

Multi-Head Attention 是并行做多组注意力头，让不同 head 学不同关系，例如语法、指代、位置、主题。最后把多个 head 拼接再线性变换。

GPT 和 BERT 的主要差异：

- GPT 是 decoder-only，使用 causal mask，只看左侧上下文，适合生成。
- BERT 是 encoder-only，双向注意力，适合理解、分类、抽取。

### 42. SFT、RLHF、DPO 有什么区别？为什么 SFT 后还需要偏好优化？

- SFT：用高质量示范数据教模型“应该怎么答”，目标是模仿标准答案。
- RLHF：训练 reward model，再用强化学习优化模型偏好，流程复杂。
- DPO：直接用 chosen/rejected 偏好对优化，不单独训练 reward model，工程上更简单。

SFT 后还需要偏好优化，是因为示范数据只能告诉模型一个可接受答案，但无法充分表达“两个答案哪个更好”。偏好优化能强化有帮助、无害、诚实、少幻觉、遵循约束等相对偏好。

### 44. 算法题：合并两个有序链表。

思路：双指针 + 哨兵节点。

1. 新建 dummy 节点，`cur` 指向 dummy。
2. 两个指针分别遍历 `l1`、`l2`。
3. 每次把较小节点接到 `cur.Next`，对应指针后移。
4. 最后把未遍历完的链表整体接上。

时间复杂度 `O(m+n)`，空间复杂度 `O(1)`，因为只是改指针。

### 45. Java 线程池做 Agent 异步任务，参数怎么调？会遇到任务雪崩吗？

线程池参数要按任务类型调：

- CPU 密集型：线程数接近 CPU 核数，避免上下文切换。
- IO 密集型：线程数可以更大，但要看外部 API、DB、Redis、LLM provider 的承载能力。
- Agent 长任务：不要无界提交，必须有队列长度、超时、取消、限流和降级。

核心参数：

- `corePoolSize`：稳定并发能力。
- `maximumPoolSize`：突发峰值上限。
- `workQueue`：建议有界队列，避免 OOM。
- `RejectedExecutionHandler`：拒绝策略要可观测，不能静默丢任务。
- timeout：任务、工具调用、LLM 调用都要有 deadline。

任务雪崩常见于：上游流量突增、LLM 变慢、队列堆积、重试放大、定时任务集中触发。解决思路是限流、熔断、隔离线程池、指数退避、幂等去重、队列水位告警和核心链路优先级。

### 46. Synchronized 和 AQS 底层原理是什么？无锁队列在工具调用场景优势在哪？

`synchronized` 基于 JVM 对象监视器实现，进入临界区需要获取 monitor，JVM 会做偏向锁、轻量级锁、重量级锁等优化。AQS 是 Java 并发包的同步器框架，核心是 `state` + FIFO 等待队列，ReentrantLock、Semaphore、CountDownLatch 等都基于它实现。

无锁队列通常用 CAS 操作减少阻塞，适合高并发生产/消费场景，例如工具调用事件、日志事件、stream chunk 分发。优势是吞吐高、避免线程阻塞；短板是实现复杂、CAS 自旋在高竞争下也会消耗 CPU，并且要处理 ABA、内存可见性和背压。

Agent 工具调用里，我更关注有界队列和背压。无锁不是目的，避免无限堆积才是关键。

### 47. 大模型批量调用时 GC 压力怎么优化？堆外内存溢出怎么排查？

GC 压力来源通常是：大 prompt 字符串拼接、批量 JSON 序列化、日志保留完整上下文、embedding 大数组、响应缓存无上限。

优化方式：

- 控制 batch size 和最大 prompt/token。
- 流式处理响应，避免一次性把大对象放进内存。
- 复用 buffer，减少频繁分配。
- 缓存设置容量和 TTL。
- 大附件、长上下文放对象存储，只传 ID/摘要。
- 日志脱敏和截断，不记录完整大文本。

堆外内存溢出排查：

- 看 DirectBuffer、Netty buffer、mmap、JNI/native 库。
- 用 `jcmd VM.native_memory`、heap dump、GC log、容器 memory metrics。
- 检查 HTTP client 连接池、压缩/解压、向量库 SDK 是否有 native buffer 泄漏。

### 48. MySQL InnoDB 锁机制是什么？并发更新会话数据死锁怎么定位？

InnoDB 常见锁：

- 行锁：锁具体索引记录。
- 间隙锁：锁索引范围间隙，防止幻读。
- Next-Key Lock：行锁 + 间隙锁。
- 意向锁：表级锁标记，协调表锁和行锁。
- MVCC：通过 undo log 和 read view 支持一致性读。

会话数据并发更新死锁常见原因：

- 多事务更新同一批消息/会话，但顺序不一致。
- 缺索引导致范围锁扩大。
- 先查后写范围太大。
- 长事务里夹杂外部 API/LLM 调用。

定位方式：看 `SHOW ENGINE INNODB STATUS` 的 latest deadlock、慢 SQL、事务持锁时间、执行计划。修复原则是统一更新顺序、补索引、缩短事务、乐观锁版本号、把 LLM/工具调用放到事务外。

### 48A. 数据库索引失效有哪些情况？LIKE 查询会不会导致索引失效？

索引失效的本质是查询条件无法利用 B+Tree 的有序性，或优化器判断走索引不划算。

常见情况：

- 对索引列做函数或表达式：`DATE(created_at) = ...`、`LOWER(name) = ...`。
- 隐式类型转换：字符串列拿数字比较，可能导致索引列被转换。
- 联合索引不满足最左前缀：索引 `(a, b, c)`，只按 `b` 查通常用不上。
- 范围条件后面的列利用受限：`a = ? AND b > ? AND c = ?` 中 `c` 可能无法继续用于范围定位。
- `OR` 两边不是都有可用索引，优化器可能全表扫。
- `!=`、`NOT IN`、`IS NOT NULL` 选择性差时可能不走索引。
- 前导模糊 LIKE：`LIKE '%abc'` 或 `LIKE '%abc%'` 不能利用普通 B+Tree 前缀。
- 排序字段和过滤字段不匹配，导致 filesort 或回表成本过高。
- 低选择性字段，例如性别、状态，单独建索引收益有限。

LIKE 要分情况：

- `LIKE 'abc%'` 可以用前缀索引。
- `LIKE '%abc'`、`LIKE '%abc%'` 通常不能用普通索引。
- 全文搜索应考虑 fulltext、Elasticsearch/OpenSearch 或 trigram，而不是硬靠 B+Tree。

排查时用 `EXPLAIN` 看 type、key、rows、Extra，再结合慢 SQL 和真实数据分布判断。面试里不要绝对说“某写法一定失效”，优化器会根据统计信息和成本选择执行计划。

### 49. 大模型特征缓存怎么解决热点 Key 和缓存污染？

特征缓存可以缓存 embedding、query rewrite、rerank 特征、用户 profile 摘要等。问题是热点 key 会被大量请求打爆，低价值 key 会污染缓存。

解决：

- 热点 key：singleflight/互斥重建、本地缓存 + Redis、多级缓存、逻辑过期、预热。
- 缓存污染：设置准入策略，只缓存高频/高成本结果；LRU + LFU；短 TTL；按租户/业务分区。
- Key 设计：包含模型版本、prompt 版本、权限范围、数据版本，避免错用旧缓存。
- 监控：命中率、回源率、key 分布、value 大小、重建耗时。

### 50. Redis 存长对话有什么隐患？

Redis 适合缓存和短期状态，不适合无限保存长对话全文。隐患包括：

- 内存成本高，长 value 容易造成大 key。
- 淘汰策略可能导致状态丢失。
- 大 value 读写阻塞单线程。
- 持久化和复制压力增加。
- 敏感数据长期放缓存有合规风险。

更稳的方案是：Postgres/对象存储作为事实源，Redis 存 session 热状态、游标、短摘要、job event；长对话通过压缩、分页、检索按需加载。

### 51. Kafka 消息堆积时，怎么避免 Agent 定时任务漏执行、重复执行？

先区分“至少一次”和“恰好一次”。大多数业务按至少一次设计，再用幂等保证不会重复产生副作用。

做法：

- 消费端水平扩展，增加 partition 和 consumer。
- 监控 consumer lag、处理耗时、失败率。
- 定时任务写入任务表或 outbox，状态从 pending -> running -> success/failed。
- 用业务幂等 key 防重复执行，例如 `tenant_id + job_type + scheduled_at`。
- ACK 在副作用完成后再提交。
- 超时 running 任务由 watchdog 重新调度。
- 对高风险工具调用记录 call ID，重试前先查是否已完成。

### 52. 工程层面怎么系统性降低大模型幻觉？不要只聊 Prompt。

系统性做法：

- 数据层：RAG evidence 有来源、时间、权限和可信度。
- 检索层：多路召回 + rerank + citation，低置信度不强答。
- 工具层：事实问题优先查 API/DB，而不是模型猜。
- 生成层：结构化输出、温度控制、引用约束。
- 校验层：groundedness 检查、规则校验、敏感动作审批。
- 观测层：把幻觉 badcase 写入 golden set，持续回放。

一句话：Prompt 是软约束，数据、工具、校验、权限、评测才是硬约束。

### 53. RAG 召回率差，除了调 embedding，检索链路还有哪些优化点？

可以从链路逐段排查：

- 文档解析：PDF、表格、代码块是否解析错。
- Chunk：是否切太碎、太长、缺标题路径。
- Metadata：权限、时间、类型过滤是否误杀。
- Query rewrite：是否抽取了正确实体、同义词、缩写。
- 多路召回：BM25、向量、FAQ、热门、最近内容是否组合。
- Rerank：是否把正确结果排上来。
- 索引质量：embedding 模型版本、维度、collection 是否一致。
- 评测：gold evidence 是否标注准确。

如果 BM25 已经能命中，向量检索可以作为补召回，不一定每个 query 都等权融合。

### 54. 向量数据库和 Redis 存向量，高并发下各自瓶颈在哪？

Redis 存向量适合小规模、低延迟、简单缓存，但瓶颈是内存成本、单线程命令执行、大 key、缺少专业 ANN 索引和复杂过滤能力。

向量数据库适合大规模 ANN 检索和 metadata filter，但瓶颈是：

- embedding 生成延迟。
- ANN 查询 CPU/内存。
- 分片和副本带来的网络开销。
- payload filter 过复杂导致查询变慢。
- upsert/delete 高频更新影响索引。

高并发方案：embedding 缓存、读写分离、按租户/业务分 collection 或 shard、热点 query 缓存、限流、批量 upsert、召回 topK 控制。

### 55. 缩放点积注意力为什么除以 `sqrt(d)`？推理时怎么优化多头注意力？

`QK^T` 的方差会随维度 `d` 增大而变大，不除以 `sqrt(d)` 时 softmax 容易饱和，梯度和注意力分布不稳定。缩放后能让分数范围更稳定。

推理优化：

- KV cache：历史 token 的 K/V 不重复计算。
- MQA/GQA：多个 query head 共享 K/V，减少显存和带宽。
- FlashAttention：优化 attention 的 IO 和显存访问。
- 量化：降低权重和 KV cache 精度。
- Continuous batching：动态合批提升 GPU 利用率。
- 限制上下文和输出长度：直接降低计算量。

### 56. Function Call 失败率高，怎么设计三级容错回退？

可以设计三级：

1. 参数修复：schema 校验失败，把错误信息返回给模型，让模型只修 JSON 参数。
2. 工具降级：主工具失败时换备用工具、缓存结果、只读查询或人工确认。
3. 用户澄清/人工介入：连续失败、低置信度或高风险操作时停止自动执行。

每一级都要记录 trace：原始参数、校验错误、重试次数、最终状态。重试要有上限，避免无限 tool loop。

### 57. 上下文窗口超限时，滑动窗口和摘要压缩各有什么硬伤？

滑动窗口：

- 优点：简单、保留最近细节。
- 硬伤：早期关键决策、用户约束、工具结果可能被丢掉。

摘要压缩：

- 优点：能保留长程信息，节省 token。
- 硬伤：摘要可能丢细节、引入幻觉、把临时错误固化。

更好的方案是混合：保留最近窗口 + 关键事实 memory + 可追溯摘要 + 原始消息按需检索。摘要进入长期 memory 前要有来源和置信度。

### 58. LangGraph 状态机有什么性能短板？自研框架怎么优化？

LangGraph 的短板可能在：

- 节点多时状态序列化和 checkpoint 开销大。
- Python runtime 和 callback 链路带来额外延迟。
- 大 State 传递导致内存和网络开销。
- 高并发下 checkpointer/存储成为瓶颈。

自研优化：

- State 瘦身，只传 ID、摘要和必要字段。
- checkpoint 增量化，不每步写完整大状态。
- 热路径用 Go/Java 服务承载，LLM/工具作为外部节点。
- 异步事件流和批处理。
- 节点级 timeout、熔断、trace。

### 59. 大模型接口频繁抖动，怎么设计熔断不影响核心流程？

核心思路是隔离和降级：

- 按 provider/model 维护健康状态和失败计数。
- 对 timeout/5xx/限流做重试和指数退避。
- 连续失败触发熔断，短时间不再打这个 backend。
- 自动切备用模型或小模型。
- 非核心能力降级，例如关闭自动总结、延迟 memory 抽取。
- 核心链路返回可解释状态，而不是卡死。

本项目里的 LLM governance 就是这个方向：记录失败率、latency、TTFT、quota，并支持多 backend 路由。

### 60. 分布式场景下，怎么保证 Agent 任务幂等，防止重复调用工具？

关键是给每个副作用动作一个幂等键：

- 任务幂等 key：`tenant_id + session_id + operation + request_id`。
- 工具调用幂等 key：`job_id + step_id + tool_name + args_hash`。
- 状态机：pending/running/succeeded/failed/cancelled。
- 执行前查调用记录，成功过就直接复用结果。
- 外部 API 支持 idempotency-key 时必须透传。
- ACK 在结果持久化后做，避免成功但状态丢失。

对删除、支付、发消息这类动作，幂等比重试更重要。

项目改造状态：

- [x] 已新增 workflow run / step 幂等字段和 `ExecuteOrResume` / `Resume`，重复请求可复用同一 workflow run，断点恢复会跳过已成功 step。
- [x] 已新增 Tool Call Ledger，工具调用 key 由 `workflow_step_id + tool_name + args_hash` 组成；成功结果可直接复用，失败可增加 attempt 重试，仍在 running 的副作用调用会进入人工复核语义。
- [x] 已支持 Memory / SQL 两类 ledger store，并在 SQL 后端通过 `agent_tool_call_ledger` 持久化。

### 61. 复杂多步骤任务中，当前 Agent 最大技术瓶颈是什么？

可以答四个方面：

- 可靠性：每一步都可能出错，错误会在长链路中放大。
- 状态管理：上下文长、工具结果多、memory 容易污染。
- 可控性：模型自主规划不稳定，高风险动作需要硬约束。
- 成本和延迟：多轮 LLM、检索、工具调用容易超预算。

解决方向是把 Agent 拆成可观测 workflow：明确状态、节点、工具权限、失败分支、checkpoint、评测和人工介入。

### 62. 从请求入参到模型返回，完整链路怎么讲？

可以按这条链路讲：

1. API 接入：认证、CSRF、租户/用户解析、rate limit。
2. 请求校验：content、附件、会话 ID、权限。
3. 会话加载：最近消息、summary、memory、附件文本。
4. 意图路由：问答、RAG、tool、job、人工审批。
5. 检索和上下文构建：hybrid search、rerank、token budget。
6. 模型调用：provider route、timeout、streaming、usage record。
7. 工具循环：tool call 校验、执行、结果回填、必要时再问模型。
8. 输出：SSE/WS 推送 delta、最终消息落库。
9. 后处理：memory 抽取、评测 trace、审计、成本统计。

### 63. 多 Agent 协作中，任务依赖和上下文污染怎么解决？举个踩坑案例。

踩坑案例可以这样讲：

> 一开始让多个 Agent 共用完整上下文，结果 reviewer 会看到 executor 的临时猜测，把未验证结论当事实继续扩散。后来改成每个子 Agent 只拿任务相关上下文和明确输入，输出必须是结构化 artifact：结论、证据、风险、未验证项。主 Agent 只合并经过验证的结果。

依赖处理：

- DAG 任务图，明确前置条件。
- 子任务输出结构化。
- 共享事实进入 memory 前要验证。
- 临时推理不写长期 memory。
- 主控统一裁决冲突。

### 64. 为什么自研调度框架不用 LangGraph？踩过哪些原生坑？

可以回答：

> LangGraph 很适合快速搭有状态 Agent，但我们的项目需要更深地接入 Go 后端、权限、SSE、job queue、审计、LLM governance、skill sandbox 和多租户隔离。如果强行把核心链路塞进 LangGraph，会出现跨语言调用、状态持久化、部署复杂度和性能调优成本。

原生坑：

- 自己要实现 checkpoint、重试、trace、节点编排。
- 状态机边界容易写散。
- 工具 schema 和权限体系需要统一抽象。
- 需要更多测试保证流程正确。

所以取舍是：原型可以 LangGraph，生产核心链路更适合贴合业务自研。

### 65. SSE 实时推送步骤进度，怎么避免大量连接把服务打崩？

SSE 是长连接，风险是连接数、慢客户端、心跳和代理超时。

设计：

- 连接数限制：按用户/租户/IP 限制。
- 心跳：定期发送 keepalive，及时清理断开连接。
- 事件缓冲：每个连接有 bounded buffer，慢客户端丢弃非关键进度。
- 横向扩展：事件写入 Redis/Kafka/pubsub，API 节点只负责 fanout。
- 断线重连：支持 last-event-id 或事件游标 replay。
- 大结果不走 SSE：只推送 artifact ID 或下载链接。

### 66. 长任务断点恢复，如何避免重启后重复执行已完成工具调用？

要把“计划步骤”和“工具调用结果”持久化：

- 每个 step 有 ID、状态和幂等 key。
- 工具调用前记录 running，完成后记录 result。
- 重启恢复时扫描 running/unknown 状态，先查外部副作用是否已完成。
- 已完成 step 不重复执行，直接复用结果。
- 对不可确认的副作用动作进入人工复核。

Checkpointer 记录流程状态，tool call ledger 记录副作用动作，两者要配合。

项目改造状态：

- [x] workflow step 已持久化 `step_index`、`attempt`、`idempotency_key` 和 metadata，恢复时按 step index 找回 checkpoint。
- [x] 工具执行层已接入 ledger，恢复同一 workflow step 时可以复用已完成工具输出，避免重复执行副作用工具。
- [x] Admin Ops 已补 workflow 恢复、取消、stale recover 和 tool call ledger 查询入口；DeepAgent 这类有完整状态的长任务可以通过恢复控制器继续执行。

### 67. 怎么设计记忆机制，区分短期记忆、长期记忆和遗忘逻辑？

- 短期记忆：当前任务、最近消息、临时工具结果，随会话结束或压缩而变化。
- 长期记忆：用户偏好、稳定事实、项目规则，需要置信度、来源、可见性。
- 工作记忆：当前多步骤任务的状态，例如待办、已完成、阻塞点。
- 遗忘逻辑：TTL、低质量归档、用户删除、冲突替换、长期未使用降权。

本项目里的 memory settings、conflict resolution、vector retrieval、context compaction 就是这套思路。

### 68. 工具调用参数解析错误，怎么实现自动纠错？

流程：

1. 模型输出 JSON/tool call。
2. 服务端 schema 校验。
3. 如果错误可修复，把错误字段、期望类型、合法枚举反馈给模型。
4. 模型只输出修正后的参数，不重新生成整段回答。
5. 二次校验仍失败则澄清或人工介入。

注意：自动纠错不能修权限问题，也不能擅自补高风险参数。

### 69. 怎么给 Agent 输出做质量打分，对抗幻觉误导业务？

可从四个维度打分：

- Groundedness：是否被 evidence 支撑。
- Correctness：关键事实是否正确。
- Completeness：是否覆盖用户问题和约束。
- Safety/Policy：是否越权、泄露、误导或执行危险动作。

工程实现：规则 + LLM judge + 人工抽检 + 用户反馈。高风险业务不能只靠 LLM judge，要有规则和人工复核。

### 70. RAG 知识库数据过时或错误，怎么规避误导 Agent？

做法：

- metadata 记录 `updated_at`、source、version、owner、可信等级。
- 检索时优先新版本、权威来源。
- 过期内容降权或不注入。
- 多来源冲突时提示不确定或走人工确认。
- 用户反馈错误后进入 review，不直接继续使用。
- 定期重建索引和抽样评测。

### 71. 怎么根据任务复杂度和成本，自动动态切换大模型？

模型路由可以按任务分层：

- 简单分类、rewrite、格式修复：小模型。
- 普通问答和摘要：中模型。
- 复杂规划、多工具、多约束推理：强模型。
- 高风险动作：强模型 + 规则 + 人工审批。

路由信号包括 query 长度、工具数量、检索失败次数、风险等级、用户等级、预算、历史成功率。失败时可升级模型，但要有最大成本和轮数限制。

### 72. MQ 批量异步任务，怎么保证执行顺序不乱？

需要先定义“哪一类顺序必须保证”。常见做法：

- 同一 session/tenant 的任务用相同 partition key。
- 单 session 内串行消费，不同 session 并行。
- 消息里带 sequence number，消费端发现缺口则等待或重排。
- 幂等状态机保证重复消息不影响结果。
- 对无顺序要求的任务拆出去并行执行。

不要为了全局顺序牺牲所有吞吐，通常只需要局部有序。

### 73. 对话敏感信息，调用模型前后怎么脱敏和还原？

流程：

1. 识别敏感信息：手机号、身份证、邮箱、地址、token、业务密钥。
2. 替换成占位符：`<PHONE_1>`、`<TOKEN_1>`。
3. 保存映射表在安全上下文，不发给模型。
4. 模型输出后只在必要位置还原，且要检查是否越权。
5. 日志和 trace 默认记录脱敏内容。

注意：不是所有内容都应该还原。对外部输出、日志、评测集要尽量保留脱敏版本。

### 74. 线上 Agent 超时暴涨，完整排查流程是什么？

排查顺序：

1. 看整体指标：QPS、P95/P99、错误率、队列长度。
2. 拆链路：认证、DB、Redis、检索、rerank、LLM、工具、SSE。
3. 看近期变更：模型版本、prompt、topK、索引、工具 API、发布。
4. 看外部依赖：LLM provider、向量库、Kafka/Redis、对象存储。
5. 看资源：CPU、内存、GC、连接池、线程池。
6. 临时止血：降级非核心能力、降低 topK、切模型、限流、扩容。
7. 复盘：把根因写入 runbook 和告警。

### 75. 项目中最棘手的故障是什么？怎么复盘根因和优化？

可以准备一个可信故事：

> 一次线上长任务超时上升，表面看是模型慢，实际根因是检索 topK 增大后 rerank 请求变多，同时部分用户附件解析产生超长 chunk，导致 prompt token 暴涨。  
> 排查时先看 P95 链路 trace，发现 LLM latency 和输入 token 同时升高，再回放具体 session，定位到附件 chunk 和 rerank 数量。  
> 优化上限制 chunk 长度、动态 topK、rerank 超时降级、token budget 截断，并把该类样本加入评测集。

重点是讲清：现象、定位证据、根因、修复、预防。

### 76. 如果重构项目，你会推翻哪些原有架构？为什么？

可以答：

- 把隐式状态改成显式状态机，便于恢复和观测。
- 把工具调用统一走 tool gateway，避免各模块私接外部系统。
- 把 memory、search、evaluation 从主对话链路解耦，异步化但可回放。
- 把多 provider 调用收敛到统一 LLM gateway，统一 timeout、quota、成本和熔断。
- 把长任务和短请求分离，长任务走 job/SSE。

不要说“全部推翻”，要说“保留领域模型，重构边界和治理能力”。

### 77. 对接企业内部业务系统，企业级 Agent 最大技术难点是什么？

难点不是模型调用，而是企业系统接入：

- 权限和数据边界复杂。
- API 老旧、不稳定、文档不全。
- 写操作需要审批、幂等、审计。
- 多租户隔离、合规、脱敏要求高。
- 工具调用错误会直接影响业务。
- 需要和现有微服务、网关、IAM、日志体系兼容。

所以企业 Agent 的核心竞争力是工具网关、权限治理、可观测、流程集成，而不是单纯 prompt。

### 78. 向量检索并发飙升，ES、FAISS 怎么分片做负载均衡？

ES/OpenSearch：

- 按租户、时间、业务域分 index。
- 合理设置 primary shard 和 replica。
- 查询走协调节点，读多场景增加 replica。
- routing key 避免跨太多 shard。
- 热数据和冷数据分层。

FAISS：

- 单机内按向量分片或 IVF index。
- 多机部署时按 hash/tenant/doc domain 分 shard。
- 查询时并行查多个 shard，再 merge topK。
- 热 shard 增加副本。
- 更新频繁的数据和静态大库分开。

Qdrant/Milvus 等向量库也类似：分片、副本、过滤字段索引、冷热分层、topK 控制。

### 79. 推理速度和准确率冲突时，怎么取舍？

先按业务分级：

- 高风险/高价值：准确率优先，允许强模型、更多检索和人工复核。
- 高频低风险：速度优先，小模型、缓存、低 topK。
- 交互式场景：先流式给进度，再后台补全。

工程上可以做 adaptive policy：根据置信度决定是否加 rerank、是否升级模型、是否多轮检索。不要全局一刀切。

### 80. 从零搭建腾讯级多租户 AI Agent 平台，架构怎么分层？

可以分八层：

1. 接入层：API Gateway、认证、租户解析、限流。
2. 编排层：workflow/agent runtime、状态机、checkpoint、job。
3. 模型层：LLM gateway、模型路由、quota、熔断、成本。
4. 工具层：tool gateway、MCP/插件、内部 API adapter。
5. 数据层：RAG、向量库、全文检索、对象存储、metadata。
6. Memory 层：短期、长期、用户偏好、项目规则、遗忘。
7. 治理层：权限、审计、风控、人工审批、脱敏。
8. 观测评测层：trace、metrics、golden set、质量评分、告警。

技术选型理由：核心链路用强类型后端承载，长任务用 MQ/job，实时反馈用 SSE/WS，检索用 hybrid search，模型通过统一 gateway 解耦。

### 81. 怎么设计通用工具网关，实现零代码对接内部业务接口？

工具网关要把内部 API 变成模型可安全调用的 tool：

- 接口注册：OpenAPI/IDL/手工配置导入。
- Schema 生成：输入输出 JSON schema。
- 权限绑定：哪些租户、用户、Agent 可调用。
- 参数映射：模型字段到内部 API 字段。
- 执行策略：timeout、重试、幂等、限流。
- 审计：记录调用人、参数摘要、结果、耗时。
- 安全：脱敏、审批、危险动作拦截。

“零代码”适合读接口和简单写接口；复杂事务仍需要人工配置流程和补偿逻辑。

### 82. 多租户下，怎么做好资源、数据、模型配额三层隔离？

- 数据隔离：所有表、索引、对象存储、memory、trace 都带 `tenant_id`；重要客户可独立库/独立 collection。
- 资源隔离：线程池、队列、连接池、worker quota 按租户限额。
- 模型配额：每日 token、请求数、成本预算、并发数、模型白名单。

还要做 noisy neighbor 防护：某个租户爆量不能拖垮全局核心服务。

### 83. 突发流量暴涨，怎么削峰限流，保证核心 Agent 功能可用？

策略：

- 接入层限流：租户/IP/用户维度。
- 队列削峰：长任务进入 MQ，短请求快速失败或排队。
- 优先级：核心客户、核心功能优先。
- 降级：关闭非核心 memory 抽取、评测、复杂 rerank。
- 缓存：热点 FAQ、embedding、检索结果。
- 熔断：外部 LLM/工具异常时切备用或返回状态。

关键是把核心问答/任务状态保住，把后台增强能力降级。

### 84. Agent 全链路监控需要哪些指标？怎么预警模型漂移、准确率下滑？

指标：

- 流量：QPS、并发、租户分布。
- 延迟：端到端、TTFT、LLM、检索、工具、队列。
- 质量：pass rate、用户反馈、LLM judge、groundedness、empty output。
- 检索：Recall@K、无结果率、rerank 分数。
- 工具：调用次数、失败率、超时率。
- 成本：token、费用、模型分布。
- 安全：越权拦截、风险事件、人工审批率。

漂移预警：固定 golden set 定期回放 + 线上反馈分桶 + 模型版本对比。如果同一批 golden set pass rate 下滑，优先怀疑模型/prompt；如果线上某类 query 下滑，优先看数据和检索。

### 85. 架构层面怎么实现 Agent 规划-执行-反思的迭代闭环？

闭环可以拆成：

1. Plan：生成任务步骤、依赖、工具选择和风险等级。
2. Execute：按步骤执行工具或检索，每步落状态。
3. Observe：收集工具结果、错误、证据和成本。
4. Reflect：判断是否完成、是否需要修正计划。
5. Stop：达到目标、预算、轮数或需要人工介入时终止。

反思不能无限循环，要有最大轮数、token budget、失败分类和人工出口。反思结果也不能直接污染长期 memory。

### 86. 小样本场景下，怎么结合提示词和 LoRA 提升 Agent 效果？

优先级：

1. 先用 prompt/few-shot 快速覆盖格式和流程。
2. 用 RAG 补知识，不把易变知识放进 LoRA。
3. 收集高质量小样本，做 LoRA 学稳定风格、工具调用格式或分类边界。
4. 用 golden set 验证 LoRA 是否真的提升，而不是过拟合。

小样本 LoRA 风险是过拟合和灾难性偏移，所以更适合“格式/风格/固定模式”，不适合频繁变化的业务知识。

### 87. Agent 如何和现有微服务、业务中台无缝兼容？

不要让 Agent 直接绕过中台。做法：

- 通过 API Gateway/IAM 使用现有身份和权限。
- Tool Gateway 封装微服务 API。
- 使用现有审计、日志、限流、熔断。
- 写操作沿用业务事务和审批。
- Agent 只作为新的交互层和编排层，不重写核心业务系统。

### 88. Serverless AI 落地 Agent 平台优缺点是什么，适合腾讯吗？

优点：

- 弹性好，适合突发和低频任务。
- 运维成本低，按量付费。
- 适合轻量工具、异步后处理、Webhook。

缺点：

- 冷启动影响交互体验。
- 长任务、长连接 SSE/WS 不一定适合。
- 本地状态和 checkpoint 要外置。
- 成本在高频长任务下可能不可控。

适合做外围工具和弹性 worker，不一定适合作为核心 Agent runtime 的唯一承载。

### 89. 2026 年 Agent 技术趋势：哪些能落地，哪些是噱头？

更能落地：

- 企业工作流里的 task-specific agents。
- 工具网关、权限治理、审计和可观测。
- 多 Agent 在代码、运维、客服、数据分析里的受控协作。
- AgentOps：评测、监控、成本、回放。
- 与办公套件、CRM、ITSM、内部中台深度集成。

偏噱头：

- 完全无人监督的通用自治 Agent。
- 没有权限和审计的“万能员工”。
- 只靠 prompt 连接生产系统。
- 不做评测就宣称端到端自动化。

面试口径：2026 的重点不是“Agent 会不会思考”，而是“能不能安全、稳定、可审计地接入生产流程”。

### 90. 对比竞品 Agent 产品，腾讯可以做出哪些差异化优势？

腾讯优势可以从场景和基础设施讲：

- 企业微信、腾讯会议、文档、邮箱、企点等办公入口。
- 云、数据库、音视频、安全、IM 的基础能力。
- 内部业务系统和行业客户生态。
- 强账号体系、权限体系和组织关系。
- ToB 场景里的私有化、合规和数据治理经验。

差异化不是再做一个聊天框，而是做“企业工作流 Agent 平台”：深接业务系统、可审批、可审计、可运营。

### 91. 异构大模型怎么统一调用格式，实现灵活调度？

做 LLM Gateway：

- 统一请求：messages、tools、response schema、stream 参数。
- 统一响应：text delta、tool call、usage、finish reason、error。
- Provider adapter：OpenAI、Claude、Gemini、Qwen、内部模型分别适配。
- 能力声明：是否支持 tool call、vision、json mode、长上下文。
- 路由策略：按任务、成本、延迟、健康状态选择模型。
- 观测：统一记录 token、latency、TTFT、错误。

本项目已有 provider 抽象和 LLM governance，可以作为类似设计。

### 92. 团队技术方案有分歧，怎么说服大家采纳你的方案？

回答结构：

1. 先对齐目标：性能、稳定性、成本、上线周期哪个优先。
2. 把方案拆成可比较维度：复杂度、风险、可回滚、可验证。
3. 做小规模 PoC 或压测，用数据说话。
4. 保留对方方案里的合理部分。
5. 决策后沉淀 ADR/规范，避免反复争论。

重点体现不是“争赢”，而是推动团队做可验证决策。

### 93. 怎么推动团队沉淀规范，避免重复踩坑？

做三类沉淀：

- 设计规范：tool schema、prompt 模板、memory 写入规则、错误码。
- 工程规范：测试、review、trace、幂等、超时、重试。
- 复盘机制：线上事故和 badcase 进入 runbook/golden set。

规范要能落到 CI、模板和脚手架里，而不是只写文档没人看。

### 94. 为什么选择腾讯做 AI Agent，而不是通用大模型公司？

可以这样答：

> 通用大模型公司更偏模型能力本身，而腾讯的优势在真实业务场景、用户入口、企业服务、云基础设施和复杂系统工程。Agent 真正落地难点不是只把模型做强，而是安全接入业务系统、处理权限、数据、审计、稳定性和成本。  
> 我更希望做的是把 Agent 变成可生产使用的业务能力，而腾讯这类平台型公司更有完整落地土壤。

## 十四、可背的项目总结版

> 这个项目的核心不是“套一个 Agent 框架”，而是把 Agent 能力产品化。  
> 后端用 Go 做了认证、会话、任务、SSE、Memory、Search、Skills、MCP、Artifact、评测、审计和部署；前端提供 Web 工作区和管理后台。  
> 检索上支持 SQL fallback、Elasticsearch/OpenSearch 全文、Qdrant 向量和 hybrid RRF 融合；Memory 上支持用户级隔离、抽取、冲突处理、向量召回、上下文注入和压缩；Agent runtime 上支持工具调用、权限、sandbox、provider 抽象、sub-agent/team 和流式输出。  
> 设计取舍是：简单任务尽量 single agent，复杂任务才拆 multi-agent；能用 workflow 固化的就不交给模型自由发挥；所有线上能力都要有超时、降级、评测和可观测。

## 十五、字节 AI 应用开发一面补充

来源：截图中的“字节 AI 应用开发一面”问题。

### 103. 讲讲企业级 SaaS 智能技术支持助手有什么技术难点，你是通过什么技术方案解决的？

企业级 SaaS 技术支持助手的难点不是简单问答，而是“准确、可控、可追溯、可接入业务系统”。

主要难点：

- 知识分散：产品文档、历史工单、FAQ、配置说明、版本变更分散在不同系统。
- 权限隔离：不同租户、角色、项目看到的知识不同，不能跨用户泄露。
- 问题表达模糊：用户经常说“上次那个报错”“控制台打不开”，需要结合上下文理解。
- 时效性强：SaaS 产品版本变更快，静态训练数据容易过期。
- 需要动作闭环：不只是回答，还要查订单、查配置、创建工单、调用内部 API。
- 答案必须可解释：企业客户需要知道依据来自哪份文档或哪条系统记录。

我的方案是 RAG + Tool + Workflow/Agent 的组合：

1. 知识侧做多路检索：文档、FAQ、历史工单、会话 memory、附件内容分别建索引。
2. 检索侧使用 hybrid：BM25/全文检索处理错误码、配置项、产品名；向量检索处理自然语言语义。
3. 权限侧在召回前后都做 `user_id`、tenant、session、visibility、status 过滤。
4. 生成侧要求引用 evidence，不知道就转人工或创建工单。
5. 动作侧通过 tool/function calling 查询业务系统，而不是让模型猜。
6. 评测侧用真实 trace、golden set、RAGAS/自定义指标持续回放。

项目里可以对应 AgentAPI 的 Search、Memory、Skills、MCP、Artifact、EvaluationEngine 和审计模块来讲，强调它是一个生产化技术支持 Agent，而不是纯 demo。

### 104. 企业级 SaaS 智能技术支持助手中问答回复准确率提升 30%，是使用什么技术方案提升的？量化标准是什么，怎么得到 30% 的提升？

回答时不要只说“调 prompt”。可以拆成 baseline、改造项和量化方式。

技术方案：

- 从纯 Prompt QA 升级为 RAG QA：引入知识库分块、embedding、向量召回和全文检索。
- 从单路向量召回升级为 hybrid 召回：BM25 处理精确词，向量处理语义问题，用 RRF 融合。
- 加入 rerank 或规则过滤：过滤低相关 chunk、过期文档、无权限内容。
- 加入 query rewrite：把口语化问题改写成检索友好的查询，同时保留原 query。
- 加入答案 groundedness 约束：要求答案基于 evidence，无法支撑时拒答或转人工。
- 加入 badcase 闭环：线上错误样本回流到评测集和知识库维护流程。

量化标准：

- 离线评测集通过率：问题是否答对，是否命中关键断言。
- Recall@K：是否召回 gold evidence。
- Groundedness：答案是否被证据支撑。
- 人工复核通过率：抽样检查自动评测偏差。
- 线上指标：用户采纳率、转人工率、二次追问率、差评率。

30% 的计算方式可以这样讲：

> 我会先固定一批业务评测集，例如 300 到 1000 条真实问题，包含 FAQ、配置、故障排查、权限、版本差异等场景。baseline 是原方案的通过率，比如 55%。改造后在同一评测集上跑，提升到 72%。相对提升是 `(72 - 55) / 55 = 30.9%`，绝对提升是 17 个百分点。面试里要说明是相对提升还是绝对提升。

如果没有真实数据，不要硬说。可以说“项目里具备这样量化的评测框架，具体 30% 需要基于固定评测集和线上反馈统计”。

### 105. 讲讲你对 RAGAS 的理解。

RAGAS 是一个 RAG 自动化评测框架，核心思路是把 RAG 质量拆成检索质量和生成质量，而不是只看最终回答。

常见指标：

- Faithfulness：答案是否忠实于检索上下文，有没有编造。
- Answer Relevancy：答案是否回答了用户问题。
- Context Precision：检索到的上下文中，靠前内容是否真的有用。
- Context Recall：应该召回的证据是否被召回。
- Context Relevancy：上下文整体和问题是否相关。

我对 RAGAS 的理解是：它适合做离线回归、badcase 对比和版本评估，但不能完全替代业务指标和人工评审。因为它通常依赖 LLM-as-judge，judge 本身会有随机性、模型偏好和领域误判。

项目里可以把 RAGAS 放在 EvaluationEngine/golden set 旁边：RAGAS 负责 RAG 维度的自动打分，自定义规则负责工具错误、延迟、成本、权限、安全等工程指标。

项目改造状态：

- [x] 已在 `EvaluationEngine.EvaluateGolden` 中落地 RAGAS 风格指标。
- [x] 指标结果会写入 `EvaluationResult.Metrics`，run 级别会聚合平均分。
- [x] 低 answer correctness、faithfulness、context recall 等会生成 finding，并触发 review。
- [x] LLM-as-Judge 已绑定线上 judge model 配置；真实评测时固定 judge model、prompt 输出 JSON schema 和 golden set version，避免分数漂移。
- [x] Admin Evaluation 页面已展示 golden run 的 RAGAS 指标，包括 answer correctness、answer relevancy、faithfulness、context precision、context recall。

### 106. 遇到过 RAGAS 打分结果不准、波动大的情况吗？可能是什么原因？怎么解决？

遇到这类问题很正常。RAGAS 波动大的原因一般有：

- Judge 模型不稳定：temperature、模型版本或 provider 更新导致分数漂移。
- 测试集太小：少量样本下单个 badcase 对均值影响很大。
- gold answer/evidence 不清晰：标准答案本身模糊，judge 难以判断。
- chunk 粒度不合理：证据被切碎或混入无关文本，影响 context precision/recall。
- 问题本身多解：不同回答都合理，但自动评测只认可一种。
- 中文或行业术语支持不稳定：judge 对专有名词、业务缩写理解不准。

解决方式：

1. 固定 judge 模型、prompt、temperature 和评测集版本。
2. 用多次评测取均值，或对关键样本做人工复核。
3. 按场景分桶看指标，不只看总体平均分。
4. 建立 golden set，明确问题、标准答案、gold evidence 和判分规则。
5. 对低置信度或争议样本进入人工评审。
6. RAGAS 只作为一个维度，同时结合 Recall@K、人工通过率、线上反馈和业务指标。

面试里可以补一句：我更信任“固定集回归趋势 + 人工抽检 + 线上反馈闭环”，不会只看一次 RAGAS 分数就判断系统好坏。

### 107. Coze / Dify 这些平台和 LangChain / LangGraph 的关系怎么理解？

可以按产品层和工程框架层区分。

Coze、Dify 更像低代码/平台化 AI 应用开发平台：

- 提供 UI 编排、知识库、插件、工作流、发布、运营和管理能力。
- 适合快速搭建客服、知识库问答、内部助手、原型验证。
- 优点是上手快、产品化程度高。
- 缺点是复杂工程定制、深度系统集成、精细权限和可观测能力可能受平台限制。

LangChain、LangGraph 更像开发框架：

- LangChain 提供 model、prompt、retriever、tool、agent 等组件抽象。
- LangGraph 更强调有状态图编排、循环、条件分支、checkpoint 和 human-in-the-loop。
- 适合工程团队构建定制 Agent runtime。

关系可以类比：

> Coze/Dify 是面向应用搭建和交付的平台，LangChain/LangGraph 是面向开发者的编排框架。平台底层可能实现了类似 LangChain/LangGraph 的能力，但给用户暴露的是更产品化的工作台。

在生产系统里，如果业务简单，可以用 Coze/Dify 快速验证；如果要深度接入企业权限、审计、队列、工具沙箱、多租户和自定义运行时，更适合自研或基于 LangGraph 类框架扩展。

### 108. 什么时候 Workflow 比 Agent 更合适？

Workflow 适合流程明确、节点稳定、可预期、对可靠性要求高的任务。

典型场景：

- 表单审核、报销审批、订单处理、工单流转。
- 数据抽取、分类、校验、入库。
- 固定 SOP 故障排查。
- 合规、安全、财务等需要确定路径和审计的流程。
- 每一步责任边界清楚，失败后需要重试或人工介入。

Agent 更适合目标明确但路径不确定的任务，例如开放式研究、代码修改、多工具探索、复杂排障。

面试口径：

> 能用 Workflow 固化的，不要轻易交给 Agent 自由发挥。Workflow 提供稳定性、可解释性和可审计；Agent 负责处理不确定性。生产系统通常是 Workflow 包住 Agent，而不是完全开放式 Agent。

### 109. Workflow 中哪些节点适合交给 LLM？

适合交给 LLM 的节点通常是“语义理解、非结构化处理、生成表达”类节点。

适合 LLM：

- 意图识别和路由。
- 非结构化文本摘要。
- 邮件、工单、聊天记录分类。
- 文档问答和 evidence 归纳。
- 信息抽取后转成结构化 JSON。
- 自然语言生成，例如客服回复、报告草稿。
- 工具参数补全，例如从用户描述中提取查询条件。

不适合直接交给 LLM：

- 金额计算、库存扣减、权限判断。
- 最终审批、删除数据、转账、发券。
- 强一致事务和不可逆操作。
- 有明确规则的字段校验。

原则是：LLM 负责理解和生成，确定性系统负责计算、校验、权限和写操作。

### 110. 人工审核点应该加在哪里？

人工审核点应该加在高风险、低置信度、不可逆和对外影响大的位置。

常见位置：

- 对外发送前：客服回复、邮件、公告、法律或合规内容。
- 写操作前：退款、封禁、权限变更、删除数据、创建生产任务。
- 低置信度时：检索证据不足、模型 judge 分数低、工具结果冲突。
- 异常分支：工具失败、外部 API 返回不一致、规则校验未通过。
- 训练数据闭环：用户差评、投诉、低分样本进入人工标注。

设计上不要只加一个“最终人工确认”。更好的方式是按风险分级：

- 低风险自动执行。
- 中风险抽样审核。
- 高风险必须人工确认。
- 不确定样本进入人工队列。

### 111. 工作流如何逐步升级成 Agentic Workflow？

Agentic Workflow 不是一步从固定流程跳到完全自治 Agent，而是逐步增加自主性。

可以分阶段：

1. 固定 Workflow：所有节点和分支由规则定义。
2. LLM 节点化：在意图识别、摘要、抽取、生成节点使用 LLM。
3. 动态路由：LLM 根据输入选择工作流分支或工具。
4. 工具调用：LLM 生成工具参数，系统校验后执行。
5. 反思和重试：失败后允许模型基于错误信息修正参数或换路径。
6. 计划执行：复杂任务先生成 plan，再按节点执行。
7. Human-in-the-loop：关键节点人工审批，审批结果反馈给后续执行。
8. 记忆和学习：把成功路径、用户偏好、badcase 沉淀为 memory/golden set。

核心原则是每增加一点自治，就要增加对应的约束：schema、权限、预算、超时、重试次数、审计和回滚。

### 113. DeepAgent 执行复杂任务时，出现无限循环、来回重复执行同一步骤，你是怎么解决的？

无限循环常见原因：

- 目标不清晰，没有明确 done condition。
- 工具返回失败但模型没有理解失败原因。
- 记忆里没有记录已尝试步骤。
- planner 每轮重新生成相似计划。
- verifier 过于宽泛，只说“不够好”，没有给具体修正方向。
- 没有预算、步数和时间限制。

解决方式：

1. 设置硬限制：最大步数、最大工具调用次数、最大 token、最大耗时。
2. 设置明确停止条件：产物存在、测试通过、字段齐全、人工确认。
3. 做步骤去重：记录 action signature，例如 tool name + args hash，重复时阻断或要求换策略。
4. 保存 task state：已尝试、已失败、已完成、当前 blocker。
5. 工具错误结构化：错误码、可重试、不可重试、建议下一步。
6. 引入 verifier：判断是否真的有进展，连续无进展就降级或转人工。
7. checkpoint + resume：失败后从最近有效状态恢复，不从头循环。

一句话总结：DeepAgent 必须有“控制器”，不能只靠模型自己决定什么时候停。

- [x] 已实现 action signature 去重：基于 `step_id + tool + args` 生成 hash，重复 action 不再执行，并累计无进展计数。
- [x] 已实现无进展阻断：Verifier 判断 action 没有推进时累计 `NoProgressCount`，超过阈值后将 DeepAgent 标记为 `blocked`，workflow run 标记为 failed。

### 114. 如何设计 DeepAgent 的记忆体系？

DeepAgent 记忆要分层，不同记忆有不同生命周期和权限。

推荐设计：

- Working Memory：当前任务的临时状态、计划、工具结果、待办和 blocker。
- Episodic Memory：历史任务过程、成功路径、失败原因、用户反馈。
- Semantic Memory：稳定知识，例如用户偏好、业务规则、产品概念。
- Procedural Memory：可复用的操作流程、工具使用经验、SOP。
- Long-term User Memory：跨会话用户偏好和长期事实。

关键机制：

1. 写入前判断价值：不是所有对话都写长期记忆。
2. 记忆要有 scope：user、session、project、organization。
3. 记忆要有状态：active、archived、conflicted、deleted。
4. 召回要结合关键词、向量、时间和权重。
5. 注入要压缩，避免把无关旧记忆塞进 prompt。
6. 冲突要处理，新事实和旧事实不一致时要标记或让用户确认。
7. 敏感信息要可删除、可审计、可关闭。

结合 AgentAPI，可以讲 Memory 服务支持用户级隔离、抽取、向量召回、上下文注入和状态治理；Live 模式也要避免 memory 触发模型主动处理旧任务。

- [x] 学习结果会写入 pending-confirm memory，由人工/后续治理确认后再进入长期记忆，避免自动污染用户记忆。

### 115. 讲讲你对 Python 协程和事件循环的理解。

Python 协程是用户态的并发机制，核心是 `async` / `await`。它适合 I/O 密集场景，比如网络请求、数据库查询、文件 I/O、WebSocket、爬虫。

事件循环可以理解为一个调度器：

1. 注册多个 coroutine/task。
2. 某个 task 遇到 `await`，主动让出控制权。
3. 事件循环去执行其他 ready task。
4. 等 I/O 完成后，再把原 task 放回 ready 队列。

协程不是多线程并行。它通常在单线程内并发调度，所以 CPU 密集任务不会因为 async 自动变快，反而可能阻塞事件循环。CPU 密集任务应该用线程池、进程池或独立服务。

常见注意点：

- 不要在 async 函数里调用阻塞 I/O。
- `asyncio.gather` 可以并发等待多个任务。
- `create_task` 会把 coroutine 注册成后台任务。
- 需要处理取消、超时和异常传播。
- FastAPI、aiohttp、WebSocket 服务大量依赖事件循环模型。

面试类比到 Go：Python 协程是事件循环协作式调度；Go goroutine 是 runtime 调度的轻量线程，更适合写并发服务端，但也要处理 context、cancel、channel 和资源释放。

### 116. 做过大模型微调吗？

如果没有完整生产微调经历，可以诚实但体现理解：

> 我没有把大模型 SFT 全流程作为生产主线落地过，更多是在 RAG、Prompt、Tool calling、评测闭环和 Agent runtime 上做工程化。但我了解微调流程，也知道什么时候该微调，什么时候不该微调。  
> 我的判断是：知识更新和私有数据优先 RAG；稳定格式、领域语气、分类边界、工具调用习惯这类问题，可以考虑微调。微调前必须先有高质量数据和评测集，否则只是把噪声训练进模型。

如果要说做过实验，可以补：

> 实验层面会用开源模型做 LoRA/SFT，准备 instruction 数据，划分 train/validation/test，用 loss、自动指标和人工集评估，再和 prompt/RAG baseline 对比。

### 117. 训练集、验证集、测试集如何划分？作用分别是什么？

三者作用不同：

- 训练集：用于更新模型参数，让模型学习任务分布。
- 验证集：用于调超参、选 checkpoint、早停和比较方案。
- 测试集：最终只用一次或少量使用，用来评估泛化能力。

常见划分比例：

- 数据量较大：80/10/10 或 70/15/15。
- 数据量较小：可以用交叉验证，但最终仍要保留独立测试集。

划分原则：

- 按用户、时间、业务场景分层，避免数据泄漏。
- 同一个用户、同一个工单、同一份文档的高度相似样本不要同时出现在训练和测试中。
- 测试集要覆盖真实线上分布和关键长尾场景。
- 对 Agent/RAG 任务，还要区分检索样本、生成样本、工具调用样本和多轮样本。

验证集是开发过程中反复看的，测试集是最后验收用的。如果根据测试集不断调参，测试集就被污染了。

### 118. 如果让你来对大模型微调，你会怎么做？讲讲思路。

我会按目标、数据、训练、评估、上线五步做。

1. 明确目标

先判断微调解决什么问题：

- 输出格式不稳定。
- 工具调用参数不稳定。
- 特定领域分类或抽取。
- 固定语气和客服话术。
- 小模型替代大模型降低成本。

如果问题是知识过期或私有知识问答，我会优先用 RAG，不优先微调。

2. 准备数据

- 收集真实高质量样本，去重、脱敏、清洗。
- 统一 instruction/input/output 格式。
- 标注拒答、边界条件、错误示例和困难样本。
- 按场景分桶，保证训练集覆盖主场景和长尾。
- 划分 train/validation/test，避免同源泄漏。

3. 选择方法

- 小规模风格/格式适配：LoRA/QLoRA。
- 分类、抽取：SFT 通常足够。
- 偏好对齐：可以考虑 DPO/RLHF，但成本更高。
- 如果已有强模型且只是工具调用格式问题，优先 function calling/schema 约束，不一定微调。

4. 训练和调参

- 选择 base model，确定 context length、learning rate、batch size、epoch。
- 监控 train loss 和 validation loss，防止过拟合。
- 保存多个 checkpoint，用验证集选最优。
- 对安全、拒答、越权场景做专项评测。

5. 评估和上线

- 和 prompt/RAG baseline 对比，而不是只看 loss。
- 看准确率、格式通过率、工具调用成功率、幻觉率、延迟和成本。
- 先灰度上线，保留 fallback 模型。
- 线上 badcase 回流到数据集，形成持续迭代。

面试总结：

> 微调不是第一反应。我的顺序是先用 prompt/schema/tool/RAG 解决可工程化的问题；当任务分布稳定、数据质量足够、格式或行为需要模型内化时，再做 SFT/LoRA，并且必须配套独立评测集和回滚方案。

## 十六、`agentruntime` 源码扫描补充：当前项目可讲模块

本节根据 `internal/backend/agentruntime` 当前代码扫描整理，用来把“项目怎么设计的”讲得更贴近真实代码。面试时不用逐文件背，可以按模块讲架构、关键能力、源码证据和可补点。

### 1. Runtime 总体分层怎么讲？

`Runtime` 是 AgentAPI 的业务运行时门面，代码在 `runtime.go`。它不是一个单纯 chat handler，而是把会话、消息、附件、Artifact、Memory、Search、Live、Skill、Job、Workflow、Tool Ledger、DeepAgent 都收敛到统一运行时。

可以这样讲：

> Runtime 层相当于 Agent 产品的应用服务层。上面接 HTTP/SSE/WebSocket API，下面接 store、object storage、LLM runner、search、memory、skill sandbox、workflow store。它负责把一次用户请求从认证后的 API 入参，变成会话上下文、memory 注入、附件 materialize、模型调用、工具执行、消息落库、事件推送、评测和审计。

源码证据：

- `runtime.go`：`Chat`、`Live`、`CreateJob`、`RunQueuedJob`、`SearchMessages`、`ExecuteDeepAgentTask`、`ResumeWorkflowRun`。
- `routes.go` / `server.go`：普通用户 API 与 Admin Ops API。
- `store_sql.go`、`workflow_sql.go`、`tool_call_ledger.go`：生产持久化边界。

### 2. Live API 现在怎么设计？

Live 模块在 `live.go`、`live_cache.go`、`live_noise.go`、`live_token.go`。当前是后端代理 Vertex/Gemini Live WebSocket，前端把音频和 activity event 发给后端，后端再连接上游 Live API。

关键设计点：

- `VertexLiveService.Run` 负责上下游 WebSocket 转发，向前端发送 `live_ready`、`live_setup_complete`、`live_response_end`、`message`、`done` 等事件。
- setup 中支持 system instruction、initial history、tool/function declarations、speech config、prebuilt voice、input/output transcription。
- 当前 VAD 策略倾向前端 activity event；当客户端发送 activity events 时，上游 provider VAD 会被关闭，避免前后端双 VAD 冲突。
- `live_noise.go` 和生成的 `live_noise_config_gen.go` 处理 ASR 噪声过滤，防止环境声被误转成德语、韩语等短句。
- `RecordLiveTurn` 会把 live turn 落回普通 session message，保证 Live 和文字对话共享上下文。
- `LiveSystemInstruction` 会通过 Prompt Registry 加载 live setup prompt，并注入 memory/技能上下文，但 prompt 明确要求：可以问候，不要在用户未要求时主动处理历史请求。

面试讲法：

> Live 不是简单前端直连模型。我们后端做了一层 Live gateway，用来统一鉴权、prompt、memory、tool、skill、transcript 持久化和噪声治理。前端负责采集音频和 VAD 活动事件，后端负责会话上下文、工具调用和上游 Gemini Live 连接，这样 Live 模式不会绕过产品里的权限、记忆和审计体系。

可继续补强点：

- [ ] Live transcript 增加语言/置信度策略，例如用户默认中文时，对短外语噪声更激进过滤。
- [ ] Live 断线恢复增加更明确的连接状态机，前端顶部监听状态可基于后端事件精确恢复。
- [ ] Live tool call 也接入完整 Tool Call Ledger，避免语音模式下重复执行副作用工具。

### 3. Workflow、断点恢复和工具账本怎么讲？

Workflow 基础在 `workflow.go`、`workflow_sql.go`；工具账本在 `tool_call_ledger.go` 和 `internal/harness/engine/tool_ledger.go`。这块可以作为“Agent 工具级幂等和断点恢复”的重点项目亮点。

当前能力：

- `WorkflowRun` 有 `request_id`、`idempotency_key`、`lease_owner`、`lease_expires_at`、`recoverable`。
- `WorkflowStepRun` 有 `step_index`、`idempotency_key`、`attempt`、`checkpoint input/output/metadata`。
- `ExecuteOrResume` 支持重复请求复用 run，`Resume` 支持从已完成 step 后继续。
- `SQLWorkflowStore` 把 run/step 写入 `agent_workflow_runs`、`agent_workflow_steps`。
- 工具调用前会构造 `ToolExecutionScope`，生成 `tool_name + args_hash + workflow step` 级别的幂等 key。
- `ToolCallLedger` 对 succeeded 结果直接复用；failed 可以重试并增加 attempt；running/unknown 这类无法确认的副作用调用进入 requires review 语义。
- Admin Ops 已有 workflow list/detail/resume/cancel/recover-stale/tool-calls 路由。

面试讲法：

> 我把长任务恢复分成两层：workflow checkpoint 解决“流程走到哪一步”，tool call ledger 解决“这一步里的副作用工具是否已经执行过”。恢复时先看 workflow step，成功的 step 不再跑；如果要重跑某一步，工具执行层再查 ledger，成功过的工具直接复用输出，无法确认的副作用工具不自动重放，进入人工复核。

源码证据：

- `WorkflowEngine.ExecuteOrResume`、`WorkflowEngine.Resume`。
- `WithToolExecutionScope`、`ToolLedgerEntry`、`BeginToolCall` / `CompleteToolCall` / `FailToolCall`。
- Admin 路由：`/v1/admin/ops/workflows/{runID}/resume`、`/tool-calls`、`/recover-stale`。

### 4. RAG/Search 现在怎么讲？

Search 相关代码包括 `message_search.go`、`message_search_infra.go`、`message_vector_index.go`、`message_attachment_index.go`、`message_search_workflow.go`。

当前能力：

- SQL fallback、Elasticsearch/OpenSearch、Qdrant semantic、hybrid search。
- Hybrid 使用 RRF 融合，后面加轻量 rerank。
- 支持 deterministic query rewrite 和低置信二轮检索，但不是无限 agentic search。
- 支持动态 TopK：模糊、多轮指代类 query 扩大 recall window。
- 消息、附件文本可进入索引；附件处理有 worker 和状态记录。
- 搜索结果 cache 已接入统一 `CacheStore`。
- `message_search_workflow.go` 把 RAG 搜索包装成 workflow step，方便观测和 checkpoint。

面试讲法：

> 检索链路不是单一向量库，而是 keyword、semantic、fallback 三层。线上先保证可用性：外部 Qdrant/Elastic 失败时不影响主对话；然后通过 RRF、rerank、动态 topK 和 query rewrite 提升召回质量。检索流程也被 workflow 化，方便追踪每次 RAG 是在哪个阶段失败。

### 5. Memory 现在怎么讲？

Memory 横跨 `store_sql.go`、`memory_governance.go`、`memory_vector.go`、`memory_workflow.go`、`memory_settings.go`、`personalization_settings.go`、`memory_assets.go`。

当前能力：

- 用户级和会话级 memory 隔离，SQL 存储按 `user_id/session_id/status` 查询。
- `LLMMemoryExtractor` + rule fallback 从 turn 中抽取 durable memory，输出 JSON 失败时有 repair。
- `applyMemoryConflictResolution` 处理冲突：用户手动 personalization 优先，冲突项可 pending/archived。
- `MemoryVectorService` 用 Qdrant 做向量召回，并和关键词/权重召回做 RRF 融合。
- memory retrieval cache 命中后仍 hydrate 当前 item，避免缓存旧内容或已删除内容。
- 支持 memory quality score、maintenance plan、merge duplicates、refresh profile、rebuild concept。
- `memory_workflow.go` 把 after-turn memory update 包装成 workflow，增强可观测。
- Asset memory extraction 可以从附件/Artifact 中抽 durable memory。

面试讲法：

> Memory 不是把历史消息全塞 prompt，而是把可长期复用的信息抽成结构化 item，并带 scope、status、confidence、source、feedback、conflict ids。注入时先检索、过滤、融合排序，再记录 injection。用户删除或关闭 memory 后，向量索引和缓存也要同步失效。

可继续补强点：

- [ ] 给 memory 写入增加更完整的人工确认 UI，尤其是低置信和冲突项。
- [ ] 增加 memory benchmark 的固定回放集，衡量抽取准确率、污染率、遗忘生效率。

### 6. Prompt Registry、A/B 和自动优化怎么讲？

Prompt 模块在 `prompt_registry.go`、`prompt_api.go`、`prompt_cache.go`、`prompt_optimization.go`。

当前能力：

- Prompt template/version/status/content hash/variables schema/render config。
- publish、rollback、diff、render preview。
- Prompt experiment：variant 权重、traffic scope、稳定分桶、winner。
- Prompt resolver cache，发布/回滚/实验变更时自动失效。
- Prompt metadata 写入 LLM usage 和 Evaluation result，便于回放“哪个 prompt 版本导致问题”。
- Prompt optimize 可以从 badcase/low-score eval 中生成候选版本，但候选需要 review/publish，不直接上线。

面试讲法：

> Prompt 不应该只存在代码常量里。生产系统要把 prompt 当成可版本化资产：有版本、有灰度、有回滚、有评测、有 usage 归因。A/B 不只看总体效果，还要看 prompt 版本在不同模型、不同场景下的失败率、成本和延迟。

### 7. LLM Governance 和 Admin Ops 怎么讲？

治理相关代码在 `llm_governance.go`、`runtime_config.go`、`server.go`、`routes.go`，后台还接了 audit、risk、quota、evaluation。

当前能力：

- `GovernedPlanner` 支持多 backend、timeout、retry、failure threshold、circuit breaker。
- 记录 LLM usage：user/session/request/skill/prompt/model/provider/token/cost/status/error/latency/TTFT。
- quota 支持 request/token/cost 维度，并支持 reset/refund adjustment。
- Admin Ops 有 health、llm-usage、llm-config、quota、audit、risk、evaluation、workflow、tool-calls 等入口。
- 最近修复了 usage 查询中 `skill_name` 为 NULL 时扫描到 string 的问题，说明普通 chat/live 调用不一定绑定 skill，治理表要按 nullable 字段处理。

面试讲法：

> LLM gateway 的价值不是只封装一个 API，而是统一模型路由、成本、配额、超时、熔断、usage 归因和后台运营。这样 prompt A/B、模型升级、成本异常和线上故障都能通过同一套 usage/trace 数据定位。

可继续补强点：

- [ ] Admin Health 中把 cache metrics、workflow recover 结果、Live 连接健康也纳入统一 snapshot。
- [ ] LLM usage 增加更多维度分桶，例如 runtime mode、tool count、retrieval backend。

### 8. Evaluation / Golden Set / RAGAS 怎么讲？

评测模块在 `evaluation_engine.go`、`evaluation_golden.go`、`evaluation_store.go`、`evaluation_api.go`、`evaluation_trace.go`、`evaluation_scheduler.go`。

当前能力：

- Runtime trace source 可从 session/job/skill/LLM usage/risk/artifact 中构建真实运行 trace。
- EvaluationEngine 统计 tool error、LLM failure、latency、token、cost、risk、empty output 等工程指标。
- Golden set 支持 version、case、expected facts、gold evidence、candidate retrieved evidence。
- RAGAS 风格指标：answer correctness、answer relevancy、faithfulness、context precision、context recall。
- LLM-as-Judge 支持 JSON 输出和 repair fallback。
- 低分/失败样本会生成 EvaluationReview，进入人工复核闭环。
- Admin API 支持 golden set CRUD、from-trace 沉淀 case、golden run、result/review/summary。

面试讲法：

> 我不会只说“准确率”。项目把 RAG 指标、工程稳定性指标和人工 review 串起来：检索看 context recall/precision，生成看 correctness/faithfulness，线上看工具错误、延迟、token、成本和风险事件。badcase 还能从 trace 回灌成 golden case，再用于 prompt/model/search 的回归。

### 9. DeepAgent 当前怎么讲？

DeepAgent 模块在 `deep_agent_controller.go`、`deep_agent_runtime.go`、`deep_agent_state.go`、`deep_agent_verifier.go`、`deep_agent_learning.go`、`deep_agent_risk.go`、`deep_agent_admin.go`。

当前能力：

- `deep_agent_task/v1` workflow：initialize、load context、plan、controller loop、verify、persist learnings。
- Planner/Executor/Verifier/Controller 接口拆分。
- Executor 支持 model、skill、rag_search 等 action。
- action signature 去重，防止重复执行同一步。
- MaxSteps、MaxActions、MaxDuration、StepTimeout、NoProgressLimit 控制无限循环。
- 高风险 action 进入 risk review pending，不直接执行。
- DeepAgent state 写入 workflow checkpoint，可 resume。
- 成功路径和 badcase 写入 `deep_agent_learnings`，再以 pending-confirm memory 形式进入长期学习候选。

面试讲法：

> DeepAgent 的难点不是让模型多想几轮，而是控制它什么时候停、什么能执行、失败后怎么恢复、学到的东西能不能进长期记忆。我的设计是 Controller 外层负责预算、状态、去重、风险和恢复，模型只负责计划和语义判断。

### 10. 安全、审计、风险和沙箱怎么讲？

相关代码包括 `risk.go`、`audit.go`、`skill_sandbox.go`、`skill_runtime_policy.go`、`skills_policy.go`、`asset_policy.go`、`web_security.go`。

当前能力：

- Audit 记录 admin 操作、prompt 发布、风险 review、skill 变更等。
- Risk scanner/review 支持高风险动作 pending、人工更新 review。
- Skill sandbox 使用 Docker，限制 network、memory、CPU、PIDs、tmpfs，skill root 只读挂载。
- Asset policy 对附件大小、类型、风险扫描做限制。
- Web 层有认证、CSRF、rate limit、安全 header。

面试讲法：

> Agent 安全不能靠 prompt。模型只能申请工具调用，真正执行必须经过服务端权限、schema 校验、sandbox、risk review 和 audit。尤其是 shell、网络、文件、发布、删除这类动作，要有硬隔离和人工出口。

### 12. 更新后的项目一句话总结

> 这个项目已经从 Web Agent demo 演进成生产化 Agent runtime：会话、消息、附件、Artifact、Live、Memory、Search、Skill、MCP、Workflow、DeepAgent、Prompt A/B、Evaluation、LLM Governance、Risk、Audit、Admin Ops 都在同一个 Go 后端里形成闭环。我的核心设计原则是：不确定性留给 Agent，确定性留给 Workflow；模型能力通过 prompt/search/memory/tool 接入，但可靠性靠 checkpoint、ledger、quota、risk review、eval 和可观测来兜底。
