## Completed P1 Scope

P1 focused on production stability, recoverability, and operational visibility.

### 1. LLM Governance

Status: Done for the first production baseline.

Delivered:

- Provider fallback chain through `-llm-fallbacks provider[:model],...`.
- Retry policy for retryable provider/transport errors with configurable attempts and backoff.
- Timeout tiers through `-llm-chat-timeout` and `-llm-skill-timeout`.
- Per-user daily quotas for successful LLM requests, tokens, and estimated cost.
- `agent_llm_usage` SQL table with user/session/request/skill/provider/model/token/cost/status/latency records.
- Model routing through `-llm-model-routes default=model,chat=model,skill=model,skill:name=model`.
- Runtime circuit breaker health state after consecutive backend failures.
- Authenticated `GET /v1/llm/status` for current backend status and governance config.

Known follow-ups:

- Replace approximate token estimates with provider-native usage when each provider exposes it through the planner layer.
- Add a non-billable or low-cost background health probe if the target provider supports it safely.
- Add monthly quota windows and user-tier-specific policies.

### 2. Durable Job Runtime

Status: Done for the first production baseline.

Delivered:

- `agent_jobs` table with queued/running/succeeded/failed/cancelled state.
- `agent_job_events` table for durable progress/result/error event replay.
- In-process worker loop for long-running chat/skill workflows.
- Authenticated job APIs: `POST /v1/jobs`, `GET /v1/jobs`, `GET /v1/jobs/{id}`, `GET /v1/jobs/{id}/events`, `POST /v1/jobs/{id}/cancel`.
- JSON event replay plus SSE replay through `GET /v1/jobs/{id}/events?stream=1&after_id=...`.
- SSE resume supports both explicit `after_id` and standard `Last-Event-ID`; stream responses include SSE `id:` fields.
- Embedded UI has a Jobs drawer with job list, event timeline replay, EventSource live updates, and job cancellation controls.
- Normal chat submission now routes long-running work to jobs internally, based on skill metadata (`run_as_job`, `long_running`, `produces_artifacts`, `execution: job`) plus conservative artifact/long-task heuristics. C-end users no longer need to choose a separate Job action.
- Online job event streams use in-process fanout after durable replay, with bounded per-client buffers and slow-client disconnect.
- Context cancellation for running jobs and final cancelled status where applicable.
- Generated artifacts carry `job_id` when created inside a job.
- User export/delete/session delete/retention cleanup include job and job event data.

Known follow-ups:

- Split the in-process worker loop into an external queue and worker service when horizontal scaling starts.
- Browser refresh, network reconnect, and multi-window viewing are recoverable through durable job/event replay, but `agentapi` process restart cannot resume an already-running in-process job. Existing job/events remain queryable after restart, but true execution recovery requires the external queue/worker phase.
- Move fanout from process memory to Redis/NATS/pub-sub when multiple API instances must share live event delivery.

### 4. Postgres Production Schema Split

Status: Done for the first production baseline.

Delivered:

- `agent_messages` table for user/assistant/tool messages with role/content/tool fields, hidden flag, payload snapshot, user/session/time indexes, and deterministic message ordering.
- `SQLSessionStore.Save` keeps `agent_sessions.payload` for compatibility and atomically replaces first-class message rows in the same transaction.
- Startup backfill copies messages from existing JSON session payloads when `agent_messages` has no rows for that session.
- User export includes a `messages` map keyed by session ID, sourced from `agent_messages` when the store supports it.
- Lightweight migration runner plus `agent_schema_migrations`; session schema v1 remains compatible, message schema is v4.
- SQL timestamp fields now use readable Postgres `TIMESTAMPTZ` instead of millisecond integers; existing `*_at` columns migrate in place while numeric fields like `latency_ms` and `size_bytes` stay numeric.
- Delete user/session and retention pruning also remove corresponding `agent_messages` rows.

Known follow-ups:

- Move all SQL stores onto the shared migration runner instead of mixed auto-create plus migration records.
- Add paginated message APIs that read directly from `agent_messages`.
- Add a production-scale background backfill/verification job for very large existing session tables.
- Split long-term memory into richer first-class rows with `kind`, `source`, `visibility`, and user-editable metadata.

### 5. Release/Deployment Hardening

Status: Done for the first production baseline.

Delivered:

- `Dockerfile.agentapi` multi-stage build for `cmd/agentapi`.
- Runtime image includes CA certificates, timezone data, curl for healthcheck, and Docker CLI for intentional Docker-sandbox deployments.
- `.dockerignore` keeps local binaries, caches, secrets, and runtime state out of image build context.
- `deploy/local/docker-compose.yml` starts Postgres, Redis, and AgentAPI with the production-oriented local baseline: SQL store, JWT user system, Cloudflare R2/S3-compatible attachment and artifact storage, Redis rate limiting, mounted local skills, health checks, and named volumes.
- `deploy/local/README.md` documents startup, health checks, provider env overrides, and local state reset.
- `deploy/production/.env.example` documents required production secrets and supported environment variables for Postgres, auth, CORS/CSRF, Cloudflare R2/S3-compatible attachment and artifact storage, Redis rate limiting, LLM governance, provider credentials, skill sandboxing, and retention.
- Graceful shutdown on `SIGINT`/`SIGTERM`: readiness flips to unavailable, live job SSE streams close, running chat/job contexts are cancelled, and the process waits up to `-shutdown-timeout` / `AGENT_API_SHUTDOWN_TIMEOUT`. Docker image declares `STOPSIGNAL SIGTERM`; local compose sets a longer stop grace period.
- `/readyz` includes Redis `PING` when Redis rate limiting is enabled, plus a non-billable LLM config/credential readiness check.
- `deploy/production/BACKUP_RESTORE.md` documents Postgres, object storage, and workspace backup/restore, local restore drills, and restore verification.

Known follow-ups:

- Replace the provider config readiness check with deeper non-billable provider
  probes where supported by each vendor.
- Add Kubernetes/Helm manifests or Terraform for the target deployment platform.
- Add automated scheduled backup jobs after the production environment is chosen.

### 2. Memory Strategy Upgrade

Status: Partial overall. The application-layer memory governance roadmap through P4 is implemented. Remaining work is intentionally deferred because it depends on production infrastructure, compliance policy, or embedding/vector decisions.

Current architecture:

- Web AgentAPI memory is implemented in `internal/backend/agentruntime`; it intentionally remains separate from the CLI/harness memory implementation under `internal/harness/memory`.
- Memory is persisted through the existing file, object-store, and SQL memory services.
- `MemoryItem` is now the primary row/document model, with schema fields for user/session scope, namespace, kind, level, category, tags, source, source references, visibility, status, content, raw hash, confidence, weight, access count, relationship/conflict links, injection trace, metadata, expiry, and timestamps.
- Long-term memory context injection selects active visible items per session using weight, relevance, quality, feedback, source priority, level boosts, and access traces.

Delivered by phase:

- Done P0/P1 baseline: first-class memory items across file/object/SQL stores with `kind`, `source`, `visibility`, `created_at`, and `updated_at`.
- Done P0/P1 baseline: user-visible memory APIs and UI for list, edit, delete, delete session memory, and delete all memory.
- Done P0/P1 baseline: explicit rule-based extraction replaces saving every turn; raw-hash dedupe, active-status filtering, confidence, weight, and basic PII/token redaction are in place.
- Done Phase 2 extraction/governance: first-class `MemoryExtractor` interface with rule, LLM, and hybrid implementations for chat turns.
- Done Phase 2 extraction/governance: centralized candidate evaluation with confidence thresholds, weak-content rejection, regex secret checks, prompt-injection filtering, sensitivity metadata, expiry hints, and content truncation.
- Done Phase 2 lifecycle: active -> dormant -> archived and expired -> deleted transitions run during retention cleanup across file/object/SQL stores.
- Done Phase 2 feedback: `POST /v1/memory/{id}/feedback` supports important, incorrect, and not-relevant signals; frontend exposes feedback controls.
- Done Phase 3 schema: atomic/concept/profile levels, parent/related IDs, conflict/supersession links, last-injected trace fields, and metadata-backed quality dimensions.
- Done Phase 3 retrieval: query-aware context selection combines stored weight, lexical relevance, source priority, concept/profile boosts, quality, feedback, recency, and usage.
- Done Phase 3 traceability: each injected item increments access count and records last injected session/query/context score.
- Done Phase 3 conflict handling: first-pass conflict detection/resolution supports pending-confirm ambiguities, archive/supersession links, and user accept/reject/keep-both resolution through `POST /v1/memory/{id}/resolve`.
- Done Phase 3 abstraction: `POST /v1/memory/rebuild` builds concept/profile memories from active atomic memories, archives stale generated abstractions, and clears dirty flags.
- Done Phase 3 maintenance: per-item quality/staleness/redundancy/usage/feedback scoring is stored in metadata.
- Done Phase 3 maintenance: rule-based organizer proposes low-quality archival, duplicate merge, dirty concept/profile rebuild, profile refresh, and pending conflict confirmation.
- Done Phase 3 maintenance: hybrid LLM/rule organizer can propose validated maintenance actions without directly mutating memory.
- Done Phase 3 maintenance APIs/UI: `/v1/memory/maintenance`, `/run`, `/{id}/apply`, `/{id}/dismiss`; Memory manager exposes Score, Run maintenance, Apply, and Dismiss.
- Done Phase 4 multimodal/source tracking: memory items support `namespace` and `source_refs` across file/object/SQL stores.
- Done Phase 4 asset extraction: `POST /v1/attachments/{id}/memory/extract` and `POST /v1/artifacts/{id}/memory/extract`.
- Done Phase 4 text assets: text/Markdown/JSON/code-like attachments and artifacts run through the same rule/LLM memory extraction pipeline as conversation memory.
- Done Phase 4 image assets: image assets attempt vision-capable extraction when the configured runner supports content blocks and fall back to explicit asset-source memory when no durable visual memory can be extracted.
- Done Phase 4 sharing controls: shared/session/user/private visibility normalization, namespace filters, source-ref filters, and namespace-aware dedupe/conflict/abstraction behavior.
- Done Phase 4 frontend: attachment/artifact rows expose Extract memory; Memory manager shows namespace, visibility, and source asset references, and lets users edit namespace/visibility.
- Done C-end governance refinement: user data export now includes itemized memory rows with metadata and `source_refs`, not only rendered user/session memory text.
- Done C-end governance refinement: PII rules cover email, phone, SSN, China ID, credit-card-like numbers, common token/password forms, and `sk-`/`pk-`/`rk-` style secrets; recoverable raw PII is still not retained.
- Done C-end governance refinement: Prometheus metrics include lightweight governance counters for data export, account delete, memory mutation/delete/maintenance, asset-memory extraction, and observed PII redactions.
- Done C-end opt-in/out controls: `MemorySettings` persist across file/object/SQL stores, `GET/PATCH /v1/memory/settings` exposes the setting, and frontend Settings lets users disable all memory, saving new memory, or using saved memory independently.
- Done C-end opt-in/out enforcement: disabled save prevents chat-turn and asset memory capture, while disabled use prevents memory context injection without deleting existing saved memory.
- Done SQL cleanup: the legacy `agent_memories` table and fallback read/delete/prune paths were removed; SQL memory now uses `agent_memory` plus `agent_memory_settings`, with startup migration from the previous `agent_memory_items` table name.

### 3. Skill Productization

Status: Done for current P2 scope. Phase 1 registry MVP, Phase 2 admin management APIs, Phase 3 version/review workflow, Phase 4 registry-backed runtime policy enforcement, and Phase 5 execution history/analytics are complete.

Delivered:

- `agent_skills` SQL registry table with name, display metadata, category/icon placeholders, status, version, source, skill root, metadata, content hash, and publish timestamps.
- Startup sync from loaded filesystem skills into `agent_skills`; new rows use the existing CLI `skill-policy` / `published-skills` settings only as bootstrap defaults.
- Registry-backed skill catalog now controls C-end visible and directly invocable skills.
- Internal Skill tool execution uses the same registry-filtered skill manager, preventing unpublished skills from bypassing publication state through model tool calls.
- `GET /v1/skills` remains the C-end skill listing API and now can surface registry-backed product metadata such as category, icon, and version when present.
- Admin-only skill registry APIs are available behind `AGENT_API_ADMIN_TOKEN` / `-admin-token` plus normal user authentication:
  - `GET /v1/admin/skills`
  - `PATCH /v1/admin/skills/{name}`
  - `GET /v1/admin/skills/{name}/versions`
  - `POST /v1/admin/skills/{name}/review`
  - `GET /v1/admin/skills/{name}/executions`
  - `GET /v1/admin/skills/{name}/analytics`
  - `POST /v1/admin/skills/{name}/publish`
  - `POST /v1/admin/skills/{name}/unpublish`
  - `POST /v1/admin/skills/{name}/disable`
- Admin publish/unpublish/disable changes refresh the runtime registry-backed catalog immediately, so new chat/job engines and the public skill list respect the latest status without restart.
- Skill registry metadata editing supports display name, description, category, icon, version, status, and metadata merge.
- Skill registry management emits audit/governance events for publish, unpublish, disable, and metadata updates.
- `agent_skill_versions` SQL table records skill version/content-hash releases with changelog and snapshot metadata.
- Startup sync records loaded skill definitions into version history without duplicating unchanged `(skill, version, content_hash)` tuples.
- Admin patch/publish accepts optional `changelog` and records version history when version/status changes.
- Admin pre-publish review checks are available through `POST /v1/admin/skills/{name}/review`; publish is rejected when blocking review issues exist, such as hidden/non-user-invocable skills, missing description, missing content hash, or inactive disabled/archived state.
- Registry metadata can now define per-skill runtime policy under `metadata.policy` or `metadata.agentapi.policy`: `allowed_tools`, `allowed_env`, `network_allowlist`, `artifact_content_types`, `shell_timeout`, and sandbox overrides for image/network/memory/CPU/pids/tmpfs/output limits.
- Per-skill `allowed_tools` and `allowed_env` are enforced when running generated skill prompts and frontmatter shell snippets.
- Per-skill `network_allowlist` restricts WebFetch/WebSearch for that skill; it is intersected with the global allowlist when a global allowlist is configured.
- Per-skill `artifact_content_types` restricts generated Artifact outputs, including wildcard families such as `image/*`.
- Per-skill shell timeout and Docker sandbox overrides are applied at runtime for skill shell execution.
- Admin Skill Management UI at `/admin` now owns the admin-token-gated policy editor for registry `metadata.policy`, covering allowed tools/env/domains, artifact content types, shell timeout, and Docker sandbox resource overrides. The C-end skill detail surface no longer exposes policy controls.
- Pre-publish review now warns when artifact-producing skills omit artifact content type policy, Bash-capable skills omit env contracts, or network-enabled sandboxes omit domain policy.
- `agent_skill_executions` SQL table records direct consumer skill execution history with user/session/job/request IDs, status, error text, duration, timestamps, and runtime policy snapshot metadata.
- Non-SQL deployments use an in-memory skill execution store for local/dev visibility.
- Direct `/skill` chat execution records success/failure and latency for both normal chat and job-routed skill work.
- Admin execution APIs expose recent execution records and aggregate totals, success/failure counts, failure rate, and average latency per skill.

### 5. Risk Control And Abuse Prevention

Status: Partial.

Implemented:

- Phase 1 request-level controls: operation-specific rate limit keys for auth login/register/refresh, chat messages, job creation, attachment upload, asset download, memory extraction, data export, account deletion, and admin actions.
- Phase 1 risk persistence: `agent_risk_events` and `agent_risk_scores` stores are available for SQL and memory backends, with admin-queryable event history.
- Phase 1 audit linkage: global and operation rate-limit denials create risk events, update abuse scores, and emit audit records where an authenticated user exists.
- Phase 2 abuse scoring: user/IP risk scores aggregate score deltas from auth failures and rate-limit/policy triggers.
- Phase 2 admin visibility: `/v1/admin/ops/risk` plus the Admin Audit/Risk Control panel show risk event queues, operation mix, and highest-risk subjects.
- Phase 3 content scanning hooks: chat/job prompts are scanned for prompt-injection, destructive command, and secret-exposure markers, then recorded into the risk event pipeline.
- Phase 3 asset scanning hooks: uploaded attachments and generated artifacts are scanned for suspicious executable extensions/content types and textual secret/malware markers, then recorded into the risk event pipeline.
- Phase 4 execution-denial hooks: product tool permission denials, skill shell sandbox command denials, sandbox network/egress failures, and sandbox resource-limit failures are classified and recorded into the same risk event pipeline.
- Phase 5 manual review queue: high-risk events and risky artifact/workflow findings are automatically queued in `agent_risk_reviews`, with Admin UI/API support for pending/in-review/resolved/dismissed handling.
- Config: `AGENT_API_OPERATION_RATE_LIMITS` / `-operation-rate-limits` supports per-operation overrides such as `chat_message=60/m`.
