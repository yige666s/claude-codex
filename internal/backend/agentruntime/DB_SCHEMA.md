# Agent Runtime Database Schema

Postgres core schema is owned by goose migrations in `migrations/postgres`.
SQLite and question-placeholder stores keep their legacy inline schema path for
local compatibility while Postgres store initialization runs the embedded goose
migrations through `RunPostgresGooseMigrations`.

## Migration Rules

1. Add every Postgres DDL change as a new numbered file in
   `internal/backend/agentruntime/migrations/postgres`.
2. Keep migrations idempotent with `CREATE TABLE IF NOT EXISTS`,
   `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, and guarded constraint blocks.
3. Do not add new destructive migrations for existing tables. Historical
   destructive versions are protected by seeding `goose_db_version` from
   `agent_schema_migrations` during legacy startup.
4. Keep `agent_schema_migrations` readable while legacy databases are still in
   the field. New Postgres migration state belongs in `goose_db_version`.

## sqlc Workflow

Postgres query code generated by sqlc lives in `dbsqlc`. Add coverage gradually:

1. Mirror the relevant table shape in `sqlc/schema/*.sql`.
2. Add named queries in `sqlc/query/*.sql`.
3. Register both files in the repository-level `sqlc.yaml`.
4. Run `make sqlc-generate` or `go generate ./internal/backend/agentruntime/dbsqlc`.
5. Use generated queries only on `SQLDialectPostgres`; keep the existing
   SQLite/question-placeholder path until support is explicitly removed.

CI should run `make sqlc-check`, which regenerates sqlc output and fails if
`sqlc.yaml`, `sqlc/`, or `dbsqlc/` has uncommitted diffs.

## Migration Tests

Postgres migration integration tests are gated by `AGENT_RUNTIME_TEST_PG_DSN`.
Each test creates a temporary schema and validates:

- fresh databases migrate from zero to the latest embedded migration;
- legacy `agent_schema_migrations` versions seed `goose_db_version`;
- repeated startup is idempotent;
- historical destructive migrations do not rerun on legacy databases.
